# Pi Review Agent 产品迭代路线

更新日期：2026-06-07

## 目标判断

当前项目已经具备 demo 发布条件：用户可以安装 pi-agent 扩展，选择内置 profile，进入卡片、直接练习、章节学习三种模式，并产生归档、总结和学习状态。下一阶段的核心不再是“能不能跑”，而是“能不能被不同学科、不同用户、不同资料质量稳定复用”。

产品定位建议保持清晰：它不是通用聊天机器人，也不是完整 LMS，而是一个基于用户资料包的复习运行时。代码负责稳定交互、状态和资料结构，agent 负责解释、出题、判题和资料整理。

## 当前产品短板

### 1. 泛用性短板

目前虽然已经支持 profile，但实际体验仍容易被 C++ 示例牵引。不同科目的资料结构差异很大：编程课有代码题和概念题，数学课有推导题，语言课有记忆和翻译题，文科课有论述和材料分析题。当前统一 schema 能覆盖基础复习，但还缺少“科目类型能力层”。

`/review-init` 的质量将决定泛用性上限。如果资料切分不稳、知识点粒度不合适、章节和卡片关系混乱，后续再好的答题 UI 也只是放大错误资料。

### 2. 可推广性短板

新用户最关心的是三件事：

- 我装完能不能立刻看到效果。
- 我的资料能不能低成本变成可复习内容。
- 出错后我能不能自己修。

当前 demo 已经解决第一点的一部分，但 README、安装脚本、错误提示和数据目录说明还不够低门槛。用户一旦遇到 `pi` 未发现命令、扩展冲突、数据保存位置不明、profile 不显示，就会把问题归因于“插件不好用”。

### 3. 学习产品短板

当前复习闭环已经存在，但还偏“单次会话”。长期学习产品需要回答：

- 今天该复习什么？
- 哪些知识点越来越弱？
- 哪些题型经常错？
- 考前应该如何安排？
- 不同 profile 的学习进度如何对比？

现在的 `state/`、`archive/`、`card_progress.json` 已经形成地基；新增的 `state/learning_profiles/{subject_id}.json` 开始把 summaries、session 和错题状态提炼成“科目学习画像”。下一步重点是把学习画像升级为今日推荐和长期复习计划。

### 4. 扩展生态短板

如果希望作为 pi-agent 插件推广，项目应尽量避免“必须理解本仓库结构才能使用”。理想状态是：

- 普通用户只知道 `/review`。
- 进阶用户知道 `/review-init` 和 `/review-fix`。
- 开发者知道 profile schema、skill 约定和工具接口。

当前这些层次还没有完全分离，README 里仍混杂开发者和普通用户信息。

## 产品演进原则

### 代码稳定，Agent 灵活

凡是需要一致交互、状态更新、路径安全、数据校验的部分，应由代码实现。凡是需要解释、变体生成、语义判断、资料重组的部分，交给 agent。

典型边界：

- 代码：profile 扫描、资料读取、卡片渲染、章节渲染、答题 UI、归档、学习状态、路径限制。
- Agent：讲解、出题、判题、追问、总结、根据用户反馈修订资料。

### Profile 是核心产品资产

不要把 profile 当作缓存文件。它应该是可迁移、可审查、可版本化、可分享的学习资料包。推广能力很大程度上取决于用户能否获取、制作和修订高质量 profile。

### Demo 不是样例代码，而是销售入口

demo profile 应展示产品最强路径：第一次进入 `/review`，用户能在 1 分钟内理解“卡片、考点、章节、答题、总结”之间的关系。demo 内容应短、稳、跨学科中性，不依赖 C++ 背景。

### 不追求一次兼容所有资料格式

PDF、Word、PPT、网页、图片笔记都会被用户期待，但不要在核心未稳时做重型导入。先把 Markdown / txt 的 profile 生成质量做高，再通过 importer 扩展输入格式。

## 路线图

### 阶段 1：发布稳定性和首次体验

目标：让普通用户安装后能稳定进入 `/review`，不会因为路径、缓存、双入口、数据目录问题卡住。

优先事项：

- README 重写为两层：普通用户快速开始、开发者说明。
- 明确推荐启动方式：安装后在任意目录运行 `pi`，或在项目根目录运行，不再推荐 `cd workspace && pi`。
- 安装脚本增加 doctor 输出：Node 版本、pi 版本、扩展入口、skills 数量、profile 数量、数据目录位置。
- `/review` 启动时显示当前数据目录和 active profile 数量，缺 profile 时自动引导 demo 或 `/review-init`。
- 对常见错误给出可执行修复提示，例如残留 `workspace/.pi`、Node 版本过低、profile schema 不合法。

验收标准：

- Windows 用户双击 `install.cmd` 后能进入 `/review`。
- `pi install git:...` 后能在任意目录启动 `/review`。
- 用户能知道 summaries、archive、state 保存在哪里。

### 阶段 2：Profile 生产质量

目标：把 `/review-init` 从“能生成资料包”升级为“能生成可复核、可修订、可启用的学习资产”。

优先事项：

- 增加 profile quality score，拆成结构完整性、知识点粒度、卡片覆盖率、章节覆盖率、考点覆盖率、source map 完整性。
- `/review-init` 生成后展示质量报告摘要，并提示用户先复核再启用。
- `/review-fix` 支持针对质量报告条目逐项修订，而不只是自由文本反馈。
- 支持 profile 导出和导入，便于同学之间共享课程资料包。
- C++ 课程资料如果面向班级公开，建议作为 bundled profile 或单独 profile 包，而不是依赖用户本地 reference。

验收标准：

- 一组普通 Markdown 笔记能生成 draft profile。
- 用户能通过 `/review-fix` 修订粒度过细、章节缺失、卡片缺失等问题。
- 启用前能看到明确质量风险。

### 阶段 3：复习推荐和长期学习

目标：从“用户选择复习什么”升级为“系统推荐今天该复习什么”。

优先事项：

- 新增 `/review-today` 或在 `/review` 首页加入“今日推荐”。
- 基于 learning profile、card progress、wrong book、knowledge chains 生成复习队列。
- 引入遗忘曲线策略，但保持可解释：未看过、最近错过、低 confidence、久未复习优先。
- session 结束后生成下一次复习建议，写入 summaries，并同步提炼到 learning profile。
- 增加按 profile 的学习仪表盘：题数、正确率、薄弱知识点、最近复习时间。

验收标准：

- 用户不选择章节也能开始一轮合理复习。
- 错题相关知识点会在后续更高优先级出现。
- 总结报告和 learning profile 都能明确给出下一步。

### 阶段 4：题型和学科适配层

目标：让不同科目的复习体验不只是“都用简答题”，而是有合理题型和判题标准。

优先事项：

- 在 profile 中增加 `subject_type`，例如 programming、math、language、humanities、general。
- 为不同 subject type 定义题型建议和判题 rubrics。
- 编程课支持代码阅读、输出判断、bug 定位、概念辨析。
- 数学课支持步骤推导、公式解释、条件变换。
- 语言课支持填空、翻译、释义、听写文本化。
- 文科课支持材料分析、论点组织、概念比较。

验收标准：

- 同一套引擎能运行至少三个不同 subject type demo。
- 每种 subject type 至少有 2-3 个稳定题型。
- 判题结果不只给对错，还能给 rubric 维度反馈。

### 阶段 5：Profile 生态和分享

目标：让 profile 成为可传播内容，而不是每个用户都从零 init。

优先事项：

- 定义 profile 包格式，例如 `.review-profile.zip` 或目录规范。
- 支持 `/review-import` 导入 profile 包。
- 支持 `/review-export` 导出 active profile，不包含个人 archive/state。
- 支持 profile metadata：作者、适用课程、版本、资料来源、更新时间、许可说明。
- 建立 examples 目录，提供多个小型 profile。
- 对班级场景，提供“课程 profile 发布者”和“学生使用者”两套说明。

验收标准：

- 一个用户生成 profile 后，另一个用户能无需 `/review-init` 直接导入使用。
- 导出的 profile 不包含个人答题记录。
- profile 更新后能通过 revision 机制替换旧版。

### 阶段 6：体验打磨和可观测性

目标：降低使用挫败感，让问题能被定位。

优先事项：

- 增加 `/review-doctor`，检查安装、路径、profile、schema、权限、pi 版本。
- TUI 页面统一信息层级：当前 profile、模式、范围、题号、正确率、session id。
- 长内容支持分页或折叠，尤其是章节学习和总结。
- 工具返回错误时统一展示用户可理解的修复建议。
- 增加 debug log，可选写入用户数据目录。

验收标准：

- 用户截图报错时，开发者能快速定位是安装问题、profile 问题、模型输出问题还是路径问题。
- 普通用户不需要读源码也能完成常见修复。

## 推广策略

### 面向同学

主打“直接复习 C++ 课程，不需要自己整理资料”。提供内置 C++ profile，安装后直接 `/review`。这类用户不应接触 `/review-init`，除非他想整理自己的课程。

推荐路径：

```text
安装 -> pi -> /review -> 选择 C++ profile -> 直接练习/章节学习
```

### 面向普通学习者

主打“把 Markdown 笔记变成复习系统”。这类用户需要清楚知道资料准备要求和 init 成本。

推荐路径：

```text
安装 -> 体验 demo -> 准备笔记 -> /review-init -> 复核 quality_report -> /review-fix -> /review
```

### 面向 pi-agent 用户

主打“pi 的学习型扩展案例”。这类用户关心 package 安装、skills、tools、profile schema 和可扩展性。

推荐路径：

```text
pi install git:... -> /review -> 查看 examples -> 自定义 profile/skill
```

### 面向开发者

主打“可复用的 agentic learning runtime”。这类用户关心边界清晰、schema、测试和扩展接口。

推荐路径：

```text
阅读 DESIGN.md -> 查看 extension tools -> 查看 profile schema -> 运行 tests -> 添加 subject adapter
```

## 推荐的近期实施顺序

1. 修正文档和启动路径，清理所有过时的 `workspace/.pi`、`cd workspace && pi`、`pi install ./workspace` 表述。
2. 增加 `/review-doctor` 或增强 `setup-review`，把常见安装错误直接诊断出来。
3. 完成 profile import/export，让班级同学不用耗 token 做 init。
4. 强化 `/review-init` 质量报告，使资料包生成可复核。
5. 做第一个非 C++ demo profile，证明泛用性。
6. 实现今日推荐队列，把长期学习价值做出来。
7. 再考虑 npm / pi 插件市场发布素材、截图、视频和更完整的项目主页。

## 暂缓事项

以下能力有价值，但不建议近期优先做：

- PDF / Word / PPT 原生导入。先通过外部工具转 Markdown。
- Web UI。当前产品依托 pi-agent TUI，过早做 Web 会分散架构。
- 云同步。先把本地数据目录和导入导出做好。
- 多人班级看板。除非明确要做教师端，否则会把产品带向 LMS。
- 复杂自定义 compaction。当前 pi-agent 自动上下文管理已经够用，复习状态应优先落到 archive/state。

## 关键风险

### Agent 输出不可控

风险：出题、判题、资料整理如果依赖自由文本，稳定性会下降。

控制方式：所有关键产物继续使用结构化 JSON 和 schema 校验；UI 不从 Markdown 正则推断题型。

### Profile 质量参差不齐

风险：用户会把资料质量问题误认为产品问题。

控制方式：质量报告必须前置，低置信度资料不能静默启用。

### 安装形态混乱

风险：本地 `.pi`、root package、workspace package、git install 多入口并存导致重复加载。

控制方式：发布形态只保留一个 canonical 入口：root package manifest -> `workspace/extensions/review/index.ts`。

### 过度 C++ 特化

风险：项目被认知为 C++ 复习脚本，而不是通用复习扩展。

控制方式：保留 C++ profile 服务班级，但新增中性 demo 和至少一个非编程 profile。

## 一句话路线

短期把安装和 demo 做稳，中期把 profile 生产和分享做强，长期把复习推荐和学科适配做成产品壁垒。
