# Pi Review Agent 后续演进指导

更新日期：2026-06-10

## 当前阶段判断

项目已经从“能跑的 pi-agent 扩展”进入“已发布后的功能迭代和质量打磨”阶段。当前基础能力包括：

- 通过 pi package 暴露 `/review`、`/review-init`、`/review-fix`。
- profile family 已落地，资料包按 `active / draft / archived / _user` 管理。
- 用户运行数据默认收敛到 `~/.pi/agent/review-data`。
- 三种复习模式已可用：卡片练习、直接练习、章节学习。
- `review_card`、`review_exam_points`、`review_chapter`、`review_answer` 等代码工具已经承担稳定交互。
- summary 和学习画像已经进入 profile 私有 `_user/` 方向。
- skill 已经完成 P0/P1 调整：`review-init` 瘦身，`review-profile-training-assets` 独立出来，卡片/考点/质量审核不再强制套固定模板。

因此，后续重点不应继续堆 prompt，而应围绕三件事演进：

1. 把数学/离散数学这类高价值场景做深。
2. 把真题和训练资产做成 profile 的一等资料层。
3. 把长期学习画像变成可执行的复习推荐。

## 产品边界

### 应坚持的定位

Pi Review Agent 是“基于资料包的复习运行时”，不是通用聊天机器人，也不是完整 LMS。

代码负责：

- profile 生命周期。
- 路径安全。
- 资料读取和结构校验。
- TUI 稳定展示。
- 答题、归档、summary、学习画像。
- 可重复的队列和状态更新。

Agent 负责：

- 解释和追问。
- 出题和判题。
- 从资料中抽取知识结构。
- 根据用户反馈修订 draft。
- 生成训练资产和题型模板。

这个边界非常重要。凡是“用户每次都要稳定看到同样交互”的内容，应尽量代码化；凡是“需要理解资料语义”的内容，可以交给 agent 和 skill。

### 不建议做的方向

- 不建议继续把 `/review-init` 做成万能导入器。init 只做基础资料包，训练资产和真题应分阶段增强。
- 不建议马上做 PDF/Word/OCR 全格式导入。先把 Markdown/txt profile 质量做高，再接 importer。
- 不建议让 agent 直接修改 active。所有资料修订仍应通过 `/review-fix` 创建 draft。
- 不建议为了数学支持直接新增大量题目 type，先用 `short_answer + rubric_hint + problem_templates` 兼容现有答题 UI。

## 近期主线

### 主线 A：离散数学资料包打磨

`lisan` 是当前最好的“数学类场景验证 profile”。它比 demo 更接近真实用户需求，也能暴露当前系统对数学训练的短板。

下一步建议先做一个 `lisan` revision draft，不直接热改 active。

优先任务：

1. 清理 `profile.json` 中旧 revision/draft 残留。
2. 重写 `quality_report.md` 和 `source_map.json` 的 subject id 与统计。
3. 处理 `cards_nt / chapters_nt / exam_points_nt` 旁路目录。
4. 合并第 1 章重复考点总结。
5. 为图论核心知识点补卡片 frontmatter 或新卡片。
6. 先为第 1-2 章生成 `problem_templates/` 试点。

验收标准：

- `/review` 选择 `lisan` 后三种模式都能顺利跑。
- 模式 1 图论知识点不会大量提示 card missing。
- 模式 2 第 1 章不会读到重复考点总结。
- `quality_report.md` 能真实反映 active/draft 状态。
- 至少 3 个数学模板可被 `review-question` 用于出题。

### 主线 B：训练资产最小闭环

当前已经新增 `review-profile-training-assets` skill，但还没有代码层和资料层的完整闭环。

首版不要追求全功能，只做最小闭环：

```text
problem_templates/
├── euclidean-algorithm.md
├── bezout-identity.md
└── linear-congruence.md
```

模板必须包含：

- 题型识别。
- 适用条件。
- 解题步骤或证明路线。
- 常见变形。
- 易错点。
- rubric。
- 出题启发式。

代码层暂时不需要新增题型。出题仍通过：

```json
{
  "type": "short_answer",
  "question_text": "...",
  "correct_answer": "...",
  "rubric_hint": "...",
  "source_basis": "problem_templates/linear-congruence.md"
}
```

验收标准：

- `review-question` 能明确引用模板作为 `source_basis`。
- 生成题不再只是“解释概念”，而能要求计算、推导、证明或步骤诊断。
- 判题时能根据 `rubric_hint` 指出具体步骤问题。

### 主线 C：真题能力分阶段落地

真题是提升复习价值的关键，但应分阶段做，不要一次性做自动切分、归类、仿真题全部能力。

建议顺序：

#### C1：真题资料结构和读取

新增资料层：

```text
past_papers/
├── raw/
├── questions/
├── past_questions.json
├── style_report.md
└── templates/
```

先手工放 3-5 道 demo 真题或模拟真题，验证代码读取。

验收：

- 能列出 profile 中的真题数量。
- 能按章节、知识点、题型过滤。
- `quality_report.md` 能统计真题覆盖率。

#### C2：真题展示和随机练习

新增工具建议：

```text
review_past_question
```

职责：

- 代码原样展示一题真题。
- 用户作答仍复用 `review_answer`。
- 归档时记录 `past_question_id`。

验收：

- `/review` 或模式 2 中能选择“真题优先”。
- 能随机抽一题真实真题。
- 真题答题记录能进入 session/archive。

#### C3：真题归类和风格分析

由 agent 读取 `past_questions.json` 和 `knowledge_index.json`，生成：

- 每题知识点映射。
- 低置信度映射列表。
- `style_report.md`。
- `templates/` 下的真题模板。

验收：

- 每道真题有 knowledge point 映射或待复核原因。
- 某知识点能列出相关真题。
- 真题风格报告能给出可执行的出题约束。

#### C4：真题风格仿真题

在直接练习模式中注入相关真题模板摘要，让 agent 生成仿真题。

验收：

- 同一知识点能生成“教材问答型”和“真题风格型”两种题。
- `source_basis` 能标明来自哪个真题模板。
- 不复制真题原题，只复用结构和考法。

### 主线 D：学习画像到今日推荐

学习画像已经有基础，但仍偏“被动注入 prompt”。下一步应变成产品入口。

建议做：

```text
/review
  -> 选择 profile
  -> 今日推荐 / 自选复习 / 查看学习画像
```

今日推荐排序因素：

- 未看过卡片。
- 最近答错知识点。
- low confidence。
- 久未复习。
- 真题高频知识点。
- 用户 summary 中的 lingering_questions。

验收：

- 空学习画像不阻塞启动流程。
- 有历史记录时，能生成 3-5 个推荐复习目标。
- 用户可以接受推荐，也可以跳过改为自选。

## 技术债优先级

### P0：影响使用的 bug

- TUI 滚动、长题干、进度条上弹。
- profile 路径/data root 混乱。
- 章节匹配错误。
- duplicate extension/tool registration。

这些必须优先于任何新功能。

### P1：影响资料包长期维护

- active 元数据残留旧 revision 链。
- archived/draft 目录膨胀但没有 doctor/prune。
- quality_report 和 source_map 过时。
- skill 运行目录和 template 源不同步。
- 卡片匹配依赖中文文件名，缺少 id/alias frontmatter。

### P2：影响推广体验

- README 混杂普通用户和开发者信息。
- 安装脚本错误信息不够可执行。
- 用户不知道数据保存在哪里。
- demo profile 不能体现数学/真题能力。

## 建议版本路线

### v3.1：数学资料包可用性版本

目标：让 `lisan` 成为高质量数学 demo。

包含：

- 整理 `lisan` revision draft。
- 生成第 1-2 章 problem templates。
- 优化 `review-question` 使用模板的 prompt。
- 补齐图论核心卡片。

不包含：

- 完整真题系统。
- 新增复杂题目 UI。

### v3.2：训练资产产品化版本

目标：让任何 profile 都能通过 `/review-fix` 增强训练资产。

包含：

- `review-profile-training-assets` 使用说明完善。
- profile quality 增加训练资产覆盖检查。
- `problem_templates/` schema 校验。
- 直接练习模式读取模板摘要。

### v3.3：真题最小版本

目标：profile 能包含真题，用户能随机抽真题做。

包含：

- `past_papers/` 目录结构。
- `past_questions.json` 读取。
- `review_past_question` 工具。
- 真题答题归档字段。
- demo profile 加 3-5 道模拟真题。

### v3.4：学习推荐版本

目标：用户打开 `/review` 后知道今天该复习什么。

包含：

- 今日推荐入口。
- 学习画像轻量展示，不阻塞。
- 复习队列生成。
- card progress、wrong book、summary 的统一推荐策略。

## 开发规则

每做一个阶段，都必须同步四类文件：

1. 运行代码：`workspace/lib`、`workspace/extensions`。
2. 运行 skill：`workspace/skills`。
3. 模板 skill：`workspace/docs/review-kit/skills`。
4. 开发文档：`workspace/docs/开发文档`。

每次提交前至少运行：

```powershell
cd workspace
npm run check
npm test
npm run check-package
npm run setup-review
```

如果改了用户数据迁移、profile family 或 archive 结构，还需要在真实数据目录上做 dry-run 诊断：

```text
~/.pi/agent/review-data
```

## 当前最推荐的下一步

不要马上做完整真题自动切分。建议先做：

1. 整理 `lisan` revision draft。
2. 为第 1-2 章生成 3 个 problem templates。
3. 让模式 2 直接练习能明确读取并引用这些模板。
4. 用一次真实复习 session 验证：题目是否真的从“概念抽查”变成“计算/证明/步骤训练”。

这条路径最短，风险最低，也最能验证产品价值。
