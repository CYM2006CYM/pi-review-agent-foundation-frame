# 14.4 分离使用和实现

[[分离使用和实现]] 是[[封装]]和[[信息隐蔽]]的重要价值之一。

对一个类来说，可以把问题分成两部分：

- 类的使用者：谁调用这个类。
- 类的实现者：这个类内部怎么组织成员函数、数据和算法。

好的类设计应让使用者依赖稳定的公开接口，而不是依赖实现细节。

## 使用者只应看到公开接口

```cpp
class Imp {
public:
    void pubFunc();

private:
    void priFunc();
    int fields;
};

class Client {
public:
    void useImp(Imp& imp)
    {
        imp.pubFunc();   // 正确：调用公开接口
        // imp.priFunc(); // 错误：私有函数不可访问
        // imp.fields = 100; // 错误：私有数据不可访问
    }
};
```

`Client` 能复用 `Imp::pubFunc`，但不知道 `pubFunc` 内部调用了哪些私有函数，也不知道 `fields` 如何组织。

如果 `Imp` 内部实现改变，只要 `pubFunc` 的接口和语义保持稳定，`Client` 就不需要修改。

## 分离的价值

| 价值 | 说明 |
|---|---|
| 复用 | 使用者直接调用已有公开接口，不重复实现 |
| 维护 | 实现内部变化不影响调用方 |
| 降低耦合 | 使用者只知道必须知道的内容 |

这也是[[软件复用]]和维护方便的基础。

> [!important] 高频考点
> 使用和实现分离的关键不是“文件上分开”，而是依赖方向上分开：使用者依赖接口，不能依赖内部实现。

## 降低使用者和实现之间的耦合

类设计时应尽量降低使用者对实现的要求。

如果一个类只能在特定平台、特定硬件、特定运行环境中使用，它的复用性就差。更好的设计是尽量减少对运行环境、平台、硬件、操作系统的直接关联。

在 C++ 中，降低耦合还包括：

- 能用前置声明时不要随意包含完整头文件。
- 能把实现放到 `.cpp` 中时，不把复杂实现写进头文件。
- 公开接口保持稳定，内部实现可以变化。

这和第 13 章的[[编译期依赖]]直接相关。

## 前置声明与外联实现

如果一个类只在声明中使用另一个类的引用或指针，通常可以用前置声明降低依赖。

但如果函数体中要调用对方成员函数，就需要完整定义。此时常见做法是把函数实现放到 `.cpp` 文件中，在 `.cpp` 中包含对应头文件。

```cpp
// A.h
class B;

class A {
public:
    void f(B& b);
};

// A.cpp
#include "A.h"
#include "B.h"

void A::f(B& b)
{
    b.func();
}
```

这样 `A.h` 对 `B.h` 的物理依赖降低，使用 `A.h` 的其他文件不必被迫包含 `B.h`。

## 本节小结

- 分离使用和实现是封装的重要目的。
- 使用者应依赖公开接口，不依赖私有实现。
- 实现变化不应迫使调用方变化。
- 降低耦合能提升复用性和维护性。
- 前置声明和外联实现可以降低编译期依赖。
