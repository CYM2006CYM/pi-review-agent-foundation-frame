# 14.5 多个类的设计

实际程序很少只靠一个类完成全部功能。更多时候，是多个类和对象相互协作完成系统功能。

多个类设计主要考虑四件事：

| 方面 | 关键问题 |
|---|---|
| [[类的拆分]] | 一个类是否承担了过多不相关职责？ |
| [[类的合并]] | 多个类是否拆得过细、互相依赖过强？ |
| [[数据关系]] | 类之间是否通过数据成员形成关联？ |
| [[行为关系]] | 类之间是否通过函数调用形成依赖与协作？ |

## 类的拆分

一个类的公开成员函数应该围绕同一个核心概念，彼此联系紧密。

如果一个类中大部分成员函数关系紧密，但有少数函数和其他功能几乎无关，就应考虑拆分。

判断线索：

- 某些成员函数很少访问其他成员。
- 某些数据只服务于一小部分行为。
- 一个类的职责变得难以用一句话说明。
- 修改某一组功能时总是牵动另一组无关功能。

拆分的目标是让每个类更清晰、更集中。

## 类的合并

拆分过细也会带来问题。

如果多个类：

- 每个都非常简单。
- 经常互相访问。
- 需要大量友元或暴露内部细节。
- 分开后反而让使用者理解困难。

就应考虑合并。

> [!warning] 易错点
> 类不是越多越面向对象。类的数量要服务于抽象清晰度，而不是为了拆而拆。

## 数据关系

多个类之间如果通过数据成员长期发生联系，通常对应第 13 章的[[关联]]。

例如一个 `Application` 需要输入、输出、配置：

```cpp
class Input;
class Output;
class Config;

class Application {
public:
    void init();
    void run();
    void term();

private:
    Input* in;
    Output* out;
    Config* cfg;
};
```

如果输入、输出、配置部分比较复杂，或者未来可能变化，就可以把它们从 `Application` 中拆出去，形成独立类。

这里的核心问题不是“能不能拆”，而是拆出去后是否让模型更清楚、变化更局部。

## 行为关系

多个类也可能通过行为发生联系，即第 13 章的[[依赖]]。

设计一个涉及多个对象的行为时，要回答两个问题：

1. 有哪些对象共同参与这个行为？
2. 这个行为是否可以分解成多个子步骤，分别交给不同类完成？

例如 `A`、`B`、`C` 三个类共同完成一个行为。理论上，这个行为可以放在 `A`、`B` 或 `C` 中。

选择原则：

- 优先放在更主动、更复杂、更容易变化的类中。
- 其他类可以作为参数传入。
- 如有必要，其他类提供委托接口，避免重复实现核心逻辑。

这与[[行为归属]]的原则一致。

## 本节小结

- 多个类设计要考虑拆分、合并、数据关系、行为关系。
- 拆分看职责是否集中，合并看类是否拆得过细。
- 数据关系通常对应关联。
- 行为关系通常对应依赖。
- 行为放在哪个类中没有绝对答案，要看主动性、复杂度和变化点。
