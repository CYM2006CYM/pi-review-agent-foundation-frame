# 14.6 单个类的设计

单个类设计要考虑的因素很多。本节从几个常见角度说明：行为可见性、数据访问、参数抽象、依赖与关联、实现抽象、数据抽象，以及类方法和实例方法的选择。

## 行为可见性

[[行为可见性]] 指一个行为是否应对类外公开。

例如 `Application` 类：

```cpp
class Application {
public:
    Application();
    void init();
    void run(Argument& arg);
    void term();

private:
    bool loadFile(string& file);
    bool saveFile(string& file);

    Input* in;
    Output* out;
    Config* cfg;
};
```

`init`、`run`、`term` 是应用程序对外可见的核心行为。`loadFile`、`saveFile` 可能只是运行过程中的内部辅助行为，不一定需要公开，因此可以放在 `private` 中。

> [!important] 高频考点
> 一个抽象行为在实现时可能拆成多个成员函数。对外可见的是接口，不是所有实现步骤。

## 数据访问：公有数据 vs getter/setter

数据成员通常设为 `private`，如果外部确实需要访问，再提供合适的 getter/setter。

getter/setter 相比直接公开数据有几个好处：

- 可以只提供 getter，不提供 setter，让数据对外只读。
- 可以在 setter 中检查合法性。
- 可以保持接口不变，内部数据表示以后仍可调整。
- 可以用 `const` 保证只读成员函数不修改对象状态。

但 getter/setter 也不应机械生成。是否提供，取决于外部是否真的需要这个接口。

## 行为参数的抽象与封装

一个成员函数除了隐含的 `this` 对象，还可能有很多外部参与对象。

例如狗对不同目标叫：

```cpp
class Dog {
public:
    void bark(Human& human);
    void bark(Human* humans, int count);
    void bark(Door& door);
    void bark(int db);
};
```

如果这些参数都表示“叫的目标”，可以进一步抽象出 `Dest`：

```cpp
class Dest;

class Dog {
public:
    void bark(Dest& dest);
};
```

这样 `Dog` 的接口更简洁，参数变化被封装到 `Dest` 中。

## 依赖还是关联

狗叫的目标可以作为参数传入：

```cpp
class Dog {
public:
    void bark(Dest& dest);
};
```

此时 `Dog` 和 `Dest` 是[[依赖]]：只有叫的时候才需要目标。

也可以让狗长期保存目标：

```cpp
class Dog {
public:
    void setDest(Dest* dest);
    void bark();

private:
    Dest* dest;
};
```

此时 `Dog` 和 `Dest` 是[[关联]]：狗对象存在期间就保存一个目标位置。

二者功能上可能都能实现，但逻辑含义不同。依赖强调临时参与，关联强调长期知道。

## 实现也可以抽象

成员函数实现很长时，可以把实现委托给另一个对象：

```cpp
class Dog {
public:
    void func(MyObj& obj)
    {
        obj.func1();
        obj.func2();
    }
};
```

这就是[[实现委托]]：把原本堆在一个函数里的复杂实现分解出去，让当前类保持清晰。

委托可以通过参数形成依赖，也可以通过数据成员形成关联。选择哪种方式，取决于委托对象是否需要长期存在于当前对象内部。

## 数据也可以抽象

数据成员不一定都是内置类型。复杂数据可以抽象成独立类。

```cpp
class DogData {
public:
    void setAge(int age);

private:
    int age;
    int weight;
    History& hist;
};

class Dog {
private:
    DogData* data;
};
```

如果一个类内部有大量相关数据，把这些数据合并为一个数据对象，往往能降低当前类复杂度。也可以继续拆成多个数据类，取决于问题域和维护需要。

## 类方法还是实例方法

最后还要考虑某些行为应该是[[类方法]]还是[[实例方法]]。

- 行为只与某个具体对象有关：用实例方法。
- 行为属于整个类或所有对象共享的规则：用类方法。

例如扑克牌背面图案属于整副牌的共同特征，修改背面图案更适合用类变量和类方法，而不是每张牌各自修改。

## 本节小结

- 单个类设计要决定哪些行为公开、哪些行为隐藏。
- 数据通常私有，必要时通过 getter/setter 暴露。
- 参数过多或概念统一时，可以抽象成参数对象。
- 依赖和关联都可表达协作，但语义不同。
- 复杂实现和复杂数据都可以进一步抽象。
- 类方法和实例方法的选择取决于行为归属对象。
