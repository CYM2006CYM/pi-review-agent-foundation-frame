# 19.2 虚机制的作用

本节从设计角度说明[[虚机制]]的价值：它不是为了“调用更复杂”，而是为了通过[[子类型化]]适应变化。

## 虚机制只是实现方式

虚机制是 C++ 实现[[动态多态]]的一种方法。

真正重要的问题不是“虚函数表怎么查”，而是：动态多态在程序设计中有什么价值。

## 保持客户代码不变

设有基类 `A`，其中声明虚函数 `func()`：

```cpp
class A {
public:
    virtual ~A() = default;
    virtual void func();
};

class B : public A {
public:
    void func() override;
};

class C : public A {
public:
    void func() override;
};
```

客户代码只依赖 `A` 的接口：

```cpp
void client(A* p) {
    p->func();
}
```

`client` 不需要知道 `p` 真实指向 `B`、`C`，还是未来新增的 `D`。只要新类继承 `A` 并重写 `func()`，客户代码就能在不修改的情况下使用新实现。

这就是虚机制的核心价值：保持访问接口不变，让实现可以变化。

## 通过子类型化适应变化

虚机制的本质不是“某个语法技巧”，而是通过子类型化组织变化。

当某个行为未来可能有多种变化形式时，可以把稳定部分放在基类接口中，把变化部分封装到不同派生类中：

```text
A::func()  定义稳定接口
B::func()  变化实现 1
C::func()  变化实现 2
D::func()  未来新增变化实现
```

新增一种变化时，通常只需要新增一个子类，而不必修改大量客户代码。

## 为什么看一行代码可能不知道结果

对于：

```cpp
p->func();
```

如果 `func` 是虚函数，并且 `p` 是基类指针，那么只看这一行代码，无法知道最终执行哪个函数体。

原因是 `p` 的动态类型可能是任何派生类，包括未来才会写出的派生类。

这种“不知道”不是缺点，而是有意的设计弹性：客户代码面向稳定接口，具体行为由对象动态类型决定。

## 面向接口编程

虚机制支持“面向接口编程”：

```cpp
void drawAll(vector<Shape*>& shapes) {
    for (Shape* s : shapes) {
        s->draw();
    }
}
```

这段代码只依赖 `Shape::draw()` 接口，不依赖 `Circle`、`Rectangle`、`Triangle` 的具体实现。未来增加 `Polygon`，只要它也是 `Shape` 的子类型，`drawAll` 就不需要改。

## 本节小结

| 角度 | 结论 |
|---|---|
| 语言机制 | 虚函数通过虚表实现动态多态 |
| 设计意义 | 客户代码依赖稳定接口 |
| 变化位置 | 派生类封装不同实现 |
| 扩展方式 | 新增子类型，而不是修改客户代码 |
| 本质目标 | 通过子类型化适应变化 |

> [!summary] 考前速记
> 虚机制的真正价值是“接口不变，实现可变”。客户代码只面向基类接口，变化封装到派生类中；新增变化时新增子类型，而不是到处修改已有调用代码。
