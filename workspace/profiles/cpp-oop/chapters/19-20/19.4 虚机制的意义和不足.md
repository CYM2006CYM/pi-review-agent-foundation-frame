# 19.4 虚机制的意义和不足

本节总结[[虚机制]]的意义，并说明它不适合直接承载多个独立变化方向。

## 虚机制的一般使用场景

一个继承树有基类和多个派生类，客户代码通过基类指针、引用或成员变量使用这棵继承树。

常见使用形式有两类：

1. [[依赖]]：函数参数是基类指针或引用；
2. [[关联]]：类中保存基类指针或引用作为数据成员。

例如：

```cpp
void func(Base& object) {
    object.f();
}
```

或者：

```cpp
class Client {
public:
    void setObject(Base* p) {
        object = p;
    }

    void run() {
        object->f();
    }

private:
    Base* object{};
};
```

不管将来新增什么派生类，只要它保持基类接口，`func` 或 `Client` 的代码就不需要修改。

## 虚机制的意义

虚机制的意义可以归纳为：

| 角度 | 意义 |
|---|---|
| 客户代码 | 面向稳定接口编程 |
| 扩展变化 | 新增派生类即可 |
| 复用性 | 客户类不用随派生类扩展而修改 |
| 维护性 | 减少改旧代码带来的风险 |

这与[[19.2 虚机制的作用]]一致：通过[[子类型化]]适应变化。

## 不足：只适合单个方向的变化

虚机制主要适合一个变化方向。

假设某个类有三个虚函数：

```cpp
class X {
public:
    virtual void f();
    virtual void g();
    virtual void h();
};
```

如果变化情况是：

- `f` 有 2 种实现；
- `g` 有 3 种实现；
- `h` 有 4 种实现。

如果全部用一个继承树承载，就要组合出：

```text
2 x 3 x 4 = 24
```

个派生类。

这会导致[[子类爆炸]]：类数量增长过快，代码高度重复，维护时容易混乱。

如果以后 `f` 从 2 种变成 5 种，类数量又会迅速增加；如果再加入新的变化方向，问题会更严重。

## 解决思路：拆分变化方向

多个独立变化方向不应该硬塞进同一棵继承树。

可以结合[[关联]]、[[依赖]]等水平关系，把每个变化方向拆成独立协作对象：

```cpp
class XF {
public:
    virtual ~XF() = default;
    virtual void f() = 0;
};

class XG {
public:
    virtual ~XG() = default;
    virtual void g() = 0;
};

class XH {
public:
    virtual ~XH() = default;
    virtual void h() = 0;
};

class X {
public:
    void f() { pf->f(); }
    void g() { pg->g(); }
    void h() { ph->h(); }
private:
    XF* pf{};
    XG* pg{};
    XH* ph{};
};
```

这样：

- `f` 的 2 种变化由 `XF` 的两个子类负责；
- `g` 的 3 种变化由 `XG` 的三个子类负责；
- `h` 的 4 种变化由 `XH` 的四个子类负责。

总共只需要：

```text
2 + 3 + 4 = 9
```

个变化类，而不是 24 个组合类。

## 本节小结

| 问题 | 处理方式 |
|---|---|
| 一个方向的变化 | 用继承和虚机制 |
| 多个方向的独立变化 | 用水平关系拆分变化方向 |
| 变化组合很多 | 避免在一棵继承树中穷举组合 |
| 客户代码复用 | 依赖稳定接口，不依赖具体实现 |

> [!summary] 考前速记
> 虚机制适合单个变化方向。多个变化方向叠加时，如果全部靠派生类组合，会出现子类爆炸。应把变化方向拆成多个协作对象，每个对象用自己的继承树处理一个方向的变化。
