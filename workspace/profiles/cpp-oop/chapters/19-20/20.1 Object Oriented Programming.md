# 20.1 Object Oriented Programming

本节从全课程角度总结[[面向对象设计过程]]：建立模型、细化模型、抽象与表示、[[封装变化]]、通过[[子类型化]]适应变化，并强调设计是[[迭代式设计]]。

## 面向对象设计的基本过程

面向对象程序设计大体包含以下阶段：

```text
建立模型
  -> 细化模型
  -> 抽象与表示
  -> 封装变化
  -> 通过子类型化适应变化
  -> 反复迭代
```

其中“建立模型”和“细化模型”更多属于面向对象分析与设计课程的内容；本课程更多关注后半部分：如何把抽象表示为类，如何封装变化，如何通过继承、虚函数和水平关系让代码适应变化。

## 建立模型：先用水平关系

建立模型时，先针对[[问题域]]抽象出核心事物及其关系。

课程强调：不要一开始就急着设计继承层次。建模初期更适合用[[关联]]和[[依赖]]这些水平关系描述事物之间的逻辑关系。

例如图书管理系统中，较高层次可以先抽象出：

- 读者或学生；
- 管理员；
- 图书管理系统。

这个模型层次很高，但过于粗略，不能直接指导实现。因此需要继续细化。

## 细化模型：降低抽象层次

细化模型仍然主要使用水平关系，把问题域中的核心类型和职责拆得更具体。

以图书管理系统为例，可以继续抽象出：

- 图书；
- 书库；
- 借书记录；
- 还书流程；
- 管理功能。

细化的目标不是立刻确定所有数据成员，而是把事物的公共行为、职责和相互关系梳理出来。

在这个阶段，领域知识非常重要。对问题域理解越深，模型细化越合理；如果对业务流程理解不够，后续设计就容易出现不周全的地方。

## 抽象与表示

抽象出类型后，需要用具体语言机制表示它们。

在 C++ 中，主要用[[类]]来表示抽象类型：

```cpp
class Book {
public:
    void borrow();
    void returnBack();
};
```

表示阶段要考虑：

- 类是否过大，是否需要拆分；
- 行为应放在哪个类中；
- 多个类是否应该合并；
- 数据如何表示；
- 数据如何组织。

例如表示一个坐标：

| 场景 | 可能表示 |
|---|---|
| 屏幕坐标 | 两个整数 |
| 平面实数坐标 | 两个浮点数 |
| 高精度坐标 | 双精度浮点数 |
| 空间坐标 | `x, y, z` 三个分量 |

数据组织也要选择数组、链表、容器等结构。

## 封装变化

面向对象设计的核心价值之一是适应未来变化，提高软件复用性和可维护性。

但初步类表示不一定是最终设计。因为：

- 设计者可能对领域知识理解不充分；
- 客户需求会变化；
- 平台、算法、数据组织方式都可能变化；
- 维护阶段会暴露新的问题。

因此需要识别可能变化的地方，把变化单独封装成类或协作对象。

例如一个类原本写成：

```cpp
class A {
public:
    void f(int n, int m);
private:
    int data[50];
};
```

这里至少有几个变化点：

- `f` 的参数列表可能变化；
- `f` 的实现算法可能变化；
- `data` 的元素类型可能变化；
- 数据组织方式可能从数组变成链表或容器。

可以把参数、数据、算法等变化点分别抽象成类：

```cpp
class Param {};
class Data {};

class A {
public:
    virtual void f(Param& p);
private:
    Data* data{};
};
```

这样变化被拆出去了，原有客户代码不必随着每个细节变化而频繁修改。

## 封装变化不是封装一切

变化非常多，不可能全部提前封装。

设计时应根据真实需求、成本、时间和风险做取舍：

- 高频变化、客户明确要求的变化，应优先封装；
- 低概率变化、成本过高的变化，可以暂时放弃；
- 设计不是追求“适应所有变化”，而是合理适应重要变化。

> [!warning] 易错点
> 面向对象设计不是把所有可能性都抽象出去。过度抽象会让系统复杂到不可维护。封装变化需要权衡。

## 通过子类型化适应变化

把某个变化点封装成独立类后，如果它只有一个主要变化方向，就可以用[[子类型化]]适应未来变化。

例如 `Data` 的组织方式可能变化：

```cpp
class Data {
public:
    virtual ~Data() = default;
    virtual void store() = 0;
};

class ArrayData : public Data {};
class ListData : public Data {};
```

未来新增 `VectorData`，只需新增子类，不必修改依赖 `Data` 接口的客户代码。

如果存在多个变化维度，应先用水平关系拆开，再让每个维度各自用继承和虚机制适应变化。

## 设计是迭代过程

面向对象设计不是一次完成的。

可能在“适应变化”阶段发现前面“封装变化”做得不好，就需要回到前面重新调整模型、职责和关系。

因此整个过程是不断迭代的：

```text
模型 -> 表示 -> 封装变化 -> 子类型化 -> 发现问题 -> 回到前面调整
```

## 本节小结

| 阶段 | 核心问题 |
|---|---|
| 建立模型 | 问题域有哪些核心事物和高层关系 |
| 细化模型 | 核心事物如何拆细，职责如何分配 |
| 抽象与表示 | 如何用类、数据和行为表达模型 |
| 封装变化 | 哪些变化点值得单独抽出 |
| 子类型化 | 如何通过派生类适应单一变化方向 |
| 迭代 | 发现设计不合理时返回前面调整 |

> [!summary] 考前速记
> 建模和细化时先用关联、依赖等水平关系，不要一上来就继承。继承和虚机制主要用于后续“适应变化”。封装变化要有选择，依据真实需求和成本权衡；设计过程是不断迭代的。
