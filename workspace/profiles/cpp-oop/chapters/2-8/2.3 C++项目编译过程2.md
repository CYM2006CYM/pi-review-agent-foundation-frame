# 2.3 C++项目编译过程2

## 本节核心

本节继续细化 C++ 项目的编译过程，重点解释[[预编译]]、[[编译]]、[[链接]]三个阶段分别检查什么、生成什么，以及不同阶段错误的区别。

> [!important] 核心认识
> 预处理负责处理 `#` 开头的预处理指令；编译阶段要求“认得每个[[标识符]]是什么”；链接阶段要求“找到每个标识符对应的实际位置”。

## 三个阶段的分工

| 阶段 | 主要任务 | 典型问题 |
|---|---|---|
| [[预编译]] / 预处理 | 处理 `#include`、`#define`、条件编译等[[预处理指令]] | 头文件展开、宏替换、条件判断 |
| [[编译]] | 检查语法和标识符含义，生成[[目标文件]] | 未声明标识符、语法错误、类型不匹配 |
| [[链接]] | 在整个程序范围内确定函数、变量等实体的位置 | 找不到函数定义、重复定义、库文件缺失 |

平时口语里说“编译通过”，常常把预处理、编译、链接三个步骤统称为“编译”。但学习底层过程时，需要把它们区分开。

## 预处理阶段处理什么

[[预编译]]也常称为预处理。它主要处理源代码中以 `#` 开头的语句。

常见[[预处理指令]]包括：

- `#include`
- `#define`
- `#ifndef`
- `#ifdef`
- `#else`
- `#endif`

其中 `#include` 只是预处理指令的一种，并不是全部。

> [!tip] 初学者理解
> 预处理阶段把源文件先加工成“编译器后面能继续处理的文本”。它还没有真正执行 C++ 程序，也没有生成最终程序。

## #include：展开头文件

例如：

```cpp
#include "My.h"

int main() {
    f(2);
    return 0;
}
```

如果 `My.h` 中有函数声明：

```cpp
void f(int);
```

那么预处理阶段会把头文件内容展开进当前 `.cpp` 对应的[[编译单元]]中，使编译器在后续编译时能够看到 `f` 的声明。

可以粗略理解为：

```cpp
void f(int);

int main() {
    f(2);
    return 0;
}
```

这解释了为什么 `.cpp` 文件中调用某个函数前，通常需要先通过头文件看到函数声明。

## #define 与条件编译

`#define` 可以定义宏名。条件编译指令可以根据某个宏是否存在，决定保留或舍弃一部分代码文本。

例如：

```cpp
#define AA

#ifndef AA
#include <iostream>
#else
void f(int);
#endif
```

这里 `#ifndef AA` 的含义是：如果没有定义 `AA`，就保留下面这一段；否则走 `#else` 部分。

因为前面已经 `#define AA`，所以 `#ifndef AA` 条件不成立，预处理会跳过 `#ifndef` 后的部分，保留 `#else` 后的内容。

> [!warning] 易错点
> `#ifndef` 判断的是宏名是否已经定义，不是判断普通变量有没有值。

## 编译阶段：要认得每个标识符

经过预处理后，编译器开始编译当前[[编译单元]]。

编译阶段的一个基本要求是：编译器必须认识当前代码中的每个[[标识符]]。

例如：

```cpp
int main() {
    f(2);
    return 0;
}
```

如果前面没有任何地方声明过 `f`，编译器就不知道：

- `f` 是函数还是变量。
- `f` 能不能被调用。
- `f` 的参数类型是什么。
- `f` 的返回值是什么。

这时会产生编译阶段错误，通常属于[[语法错误]]或语义检查错误。

## 编译阶段不要求知道实体最终地址

编译器在编译当前 `.cpp` 时，只能看到当前编译单元。

它需要知道某个名字的含义，例如：

- 这是一个变量。
- 这是一个函数。
- 它的类型是什么。
- 它的参数列表是什么。

但编译阶段不一定要求知道这个变量或函数的最终存放位置。

例如，一个函数可以在当前 `.cpp` 中声明，但在另一个 `.cpp` 中定义：

```cpp
// main.cpp
void f(int);

int main() {
    f(2);
    return 0;
}
```

只要编译器看到了 `void f(int);`，它就知道 `f` 是一个可以接收 `int` 参数的函数。至于 `f` 的函数体到底在哪个文件中，编译阶段可以先不解决。

## 链接阶段：要找到实体在哪里

[[链接]]阶段会把多个[[目标文件]]放到一起，从整个程序范围内寻找函数、变量等实体的位置。

链接阶段要解决的问题包括：

- `f` 的函数体到底在哪里。
- 某个全局变量的存储位置在哪里。
- 程序入口函数在哪里。
- 库函数或外部符号能否找到。

如果声明存在、编译通过，但链接器找不到对应定义，就会出现[[链接错误]]。

例如：

```cpp
// main.cpp
void f(int);

int main() {
    f(2);
    return 0;
}
```

如果整个工程中没有任何地方定义：

```cpp
void f(int) {
    // ...
}
```

那么编译可能通过，但链接会失败。

> [!important] 编译与链接的区别
> 编译阶段关心“这个名字我认不认识、能不能按语法使用”；链接阶段关心“这个名字对应的实际代码或存储位置能不能找到”。

## 编译错误、链接错误、逻辑错误

课程还强调：生成了可执行文件并不代表程序一定正确。

可以把错误分成三类：

| 错误类型 | 发生位置 | 典型表现 | 处理重点 |
|---|---|---|---|
| [[语法错误]] / 编译错误 | 编译阶段 | 编译器报错，无法生成目标文件或最终程序 | 按错误信息定位代码 |
| [[链接错误]] | 链接阶段 | 编译过了，但链接器找不到定义或发现重复定义 | 检查函数定义、库文件、重复定义 |
| [[逻辑错误]] | 程序运行后 | 程序能运行，但结果不对 | 调试程序逻辑和算法 |

> [!warning] 易错点
> “编译通过”不等于“程序正确”。编译和链接只能证明程序在形式上能生成可执行文件，不能保证逻辑符合题意。

## 为什么语法错误相对容易处理

初学阶段常见问题多是语法错误，例如：

- 少分号。
- 括号不匹配。
- 标识符拼写错误。
- 函数未声明。
- 参数数量或类型不匹配。

这些错误通常会被编译器直接指出。虽然错误信息可能是英文，但种类有限，随着练习增加会越来越熟悉。

更难的是[[逻辑错误]]：代码能编译、能运行，但输出结果不符合题目要求。

## 本节考点整理

| 可能题型 | 可能问法 | 答题要点 |
|---|---|---|
| 选择题 | `#include`、`#define` 属于哪个阶段处理？ | 预处理 / 预编译阶段 |
| 判断题 | `#include` 是编译阶段才处理的普通 C++ 语句。 | 错，它是预处理指令 |
| 简答题 | 编译阶段对标识符有什么要求？ | 必须知道标识符含义、类型、是否可使用 |
| 判断题 | 编译阶段一定要知道函数体最终地址。 | 错，最终地址通常由链接阶段确定 |
| 简答题 | 链接阶段主要解决什么？ | 在整个程序范围内确定变量、函数、入口等实体位置 |
| 判断题 | 只要生成可执行文件，程序逻辑就一定正确。 | 错，还可能存在逻辑错误 |
| 区分题 | 编译错误、链接错误、逻辑错误有什么区别？ | 分别发生在编译、链接、运行/结果判断阶段 |

## 本节速记

> [!summary] 速记
> 预处理处理 `#` 指令，编译要求看懂当前编译单元里的每个名字，链接要求在整个工程中找到这些名字对应的实际位置。编译通过只说明形式上能继续构建，不代表程序逻辑正确。
