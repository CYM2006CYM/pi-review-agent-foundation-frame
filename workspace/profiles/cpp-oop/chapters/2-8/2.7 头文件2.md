# 2.7 头文件2

## 本节核心

本节继续讲[[头文件]]，重点回答两个问题：

- 遇到类型相互依赖时，如何使用[[前置声明]]。
- 一个实现文件对应的头文件中，应该写哪些内容。

> [!important] 核心认识
> 头文件不是“把所有东西集中放一起”的公共仓库，而是某个模块对外公开的接口说明。

## 类型互相依赖的问题

假设有两个类型 `A` 和 `B`，并且函数声明或结构定义互相涉及它们：

```cpp
void funcFirst(A a);
void funcSecond(B b);

struct A {
    B* b;
};

struct B {
    A* a;
};
```

如果编译器从上到下读到 `void funcFirst(A a);` 时，还没有见过 `A`，它就不知道 `A` 是什么，会报错。

如果把 `struct A` 移到前面，`struct A` 内又可能先遇到 `B`。如果把 `struct B` 再移到前面，又可能先遇到 `A`。

这就是类型定义之间的[[循环依赖]]问题。

## 前置声明

[[前置声明]]可以先告诉编译器：某个名字是一个类型，但暂时不说明它内部到底有什么。

例如：

```cpp
struct A;
struct B;

void funcFirst(A* a);
void funcSecond(B* b);

struct A {
    B* b;
};

struct B {
    A* a;
};
```

或者使用类：

```cpp
class A;
class B;
```

前置声明的作用是让编译器先认识 `A` 和 `B` 这两个[[标识符]]，知道它们代表类型名。

> [!important] 关键点
> 前置声明只说明“有这个类型”，不说明这个类型的成员、大小和完整结构。

## 前置声明能解决什么

前置声明可以帮助解决：

- 两个类或结构互相引用。
- 函数参数中只需要知道某个类型名。
- 头文件之间互相包含导致结构复杂。
- 类型定义顺序难以安排。

前置声明之后，后续完整定义的顺序就更灵活。

例如：

```cpp
class A;
class B;

class A {
public:
    B* b;
};

class B {
public:
    A* a;
};
```

这里 `A` 中只保存 `B*`，编译器只需要知道 `B` 是一个类型名即可，不需要马上知道 `B` 的完整成员。

## 前置声明的限制

前置声明不是完整定义，因此不能在所有场景中替代头文件包含。

前置声明通常可以用于：

- 指针。
- 引用。
- 函数参数或返回类型的声明。

但如果需要知道对象大小或成员，就必须看到完整定义。

例如下面代码通常不够：

```cpp
class B;

class A {
    B b;  // 错误：需要知道 B 的完整大小
};
```

因为 `A` 直接包含一个 `B` 对象，编译器必须知道 `B` 的完整定义，才能确定 `A` 的对象布局。

> [!warning] 易错点
> “声明一个类型存在”和“知道这个类型内部结构”是两回事。前置声明只能解决前者。

## 为什么不能把所有声明都放进一个公共头文件

课程举了多人协作的例子：如果三个人各写一个实现文件，但所有全局变量、函数、结构声明都统一塞进一个公共头文件，表面上能用，实际会带来维护问题。

问题包括：

- 公共头文件谁负责维护不清楚。
- 某个人新增函数时，需要修改公共头文件。
- 公共头文件容易越来越大。
- 一处修改可能影响所有文件重新编译。
- 出错时责任边界不清晰。

这种写法在小程序中可能凑合，但不适合真实工程。

## 头文件应该对应模块接口

更合理的方式是：每个实现文件或功能模块有自己的头文件，用来公开自己愿意提供给外部使用的内容。

例如：

```text
my.cpp   -> my.h
your.cpp -> your.h
his.cpp  -> his.h
```

每个头文件只写本模块对外公开的内容，例如：

- 外部需要调用的函数声明。
- 外部需要使用的类型声明或定义。
- 外部需要访问的常量或配置。

不应把模块内部自用的函数和变量随便放进头文件。

## 什么应该写进头文件

头文件应包含“外部需要知道”的信息。

适合写进头文件的内容：

- 外部要调用的函数声明。
- 外部要使用的类声明或结构声明。
- 外部要使用的类型定义。
- 必须被多个编译单元共享的接口。

不适合随便写进头文件的内容：

- 只在当前 `.cpp` 内部使用的辅助函数。
- 只在当前实现文件内部使用的变量。
- 不希望外部依赖的实现细节。
- 可以隐藏在实现文件里的私有逻辑。

> [!tip] 初学者理解
> 头文件像一份“对外说明书”：别人只需要知道你提供了什么，不需要知道你内部所有临时工具。

## 内部函数不必写进头文件

如果某个函数只在当前实现文件中使用，不希望别的 `.cpp` 调用，就不应该把它声明在头文件中。

例如：

```cpp
// your.cpp
static void helper() {
    // 只给 your.cpp 内部使用
}

void publicFunction() {
    helper();
}
```

对应头文件可以只公开：

```cpp
// your.h
void publicFunction();
```

这样外部文件只能看到 `publicFunction`，看不到内部辅助函数 `helper`。

## 头文件与职责边界

从工程角度看，头文件还承担职责划分作用：

- 维护者负责自己模块的头文件和实现文件。
- 别人通过头文件了解如何使用该模块。
- 修改内部实现时，如果接口不变，外部代码不需要关心。
- 公开接口越少，模块耦合越低。

这为后续学习[[封装]]、[[信息隐藏]]和[[类和对象]]打基础。
