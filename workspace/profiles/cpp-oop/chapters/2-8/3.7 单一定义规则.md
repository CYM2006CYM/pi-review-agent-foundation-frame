# 3.7 单一定义规则

## 本节核心

本节围绕[[单一定义规则]]展开，重点是理解一个 C++ 工程中“声明可以重复，定义不能乱重复”的边界。

- 同一个[[编译单元]]里，同一个名字通常可以有多次[[声明]]，但不能有多次[[定义]]。
- 对具有[[外部连接]]的普通变量和普通函数来说，整个程序中也不能出现多个定义。
- [[包含警戒]]只能防止同一个头文件在同一个编译单元中被重复展开，不能解决多个 `.cpp` 同时包含同一份变量定义造成的链接错误。
- 头文件中通常应该放声明，普通变量和普通函数的定义应该放在实现文件中。

> [!important] 高频考点
> 初学 C++ 工程时，最常见的错误之一就是把 `int count = 1;` 这种普通全局变量定义写进头文件，然后被多个 `.cpp` 包含，最终在链接阶段出现重复定义。

## 什么是单一定义规则

[[单一定义规则]]可以先用一句话理解：

> 同一个实体在合适的范围内只能有一个定义。

这里的“实体”可以是变量、函数、类、枚举等名字。课程此处主要用变量和函数来说明。

例如在同一个作用域中：

```cpp
int count;
int count; // 错误：重复定义
```

`int count;` 对普通变量来说就是[[变量定义]]，会为 `count` 分配存储空间。重复写两次，编译器会认为同一个名字被定义了两次。

## 声明可以重复，定义不能重复

同一个函数可以多次声明：

```cpp
int add(int, int);
int add(int, int);
int add(int, int);

int add(int a, int b) {
    return a + b;
}
```

上面多次写函数声明是允许的，因为它们只是告诉编译器 `add` 的名字、参数和返回值类型。

但函数体只能定义一次：

```cpp
int add(int a, int b) {
    return a + b;
}

int add(int a, int b) { // 错误：重复定义
    return a + b;
}
```

变量也是类似：

```cpp
extern int count; // 声明：告诉编译器 count 在别处
extern int count; // 仍然是声明

int count = 0;    // 定义：真正分配存储空间
```

> [!warning] 易错点
> `extern int count;` 通常是[[变量声明]]；`int count;` 或 `int count = 0;` 通常是[[变量定义]]。把 `extern` 漏掉，就可能从“引用外部变量”变成“重新定义一个变量”。

## 头文件里放定义的问题

假设有一个头文件：

```cpp
// a.h
int count = 1;
```

如果只有一个 `.cpp` 包含它：

```cpp
// main.cpp
#include "a.h"

int main() {
    return count;
}
```

这个编译单元中只出现了一次 `count` 的定义，通常可以通过。

但如果两个 `.cpp` 都包含这个头文件：

```cpp
// a.cpp
#include "a.h"

// b.cpp
#include "a.h"
```

预处理后，两个编译单元里都会各自出现一份：

```cpp
int count = 1;
```

每个 `.cpp` 单独编译成目标文件时可能都没问题，但链接器要把多个[[目标文件]]合成一个程序时，就会发现 `count` 有多个定义，于是产生[[链接错误]]。

## 包含警戒不能解决跨文件重复定义

很多初学者会以为有了[[包含警戒]]就不会重复定义：

```cpp
// a.h
#ifndef A_H
#define A_H

int count = 1;

#endif
```

这只能保证在同一个 `.cpp` 中，`a.h` 的内容最多展开一次。

但 `a.cpp` 和 `b.cpp` 是两个不同的[[编译单元]]，它们各自都会展开一次 `a.h`。所以：

- 包含警戒解决的是“同一个编译单元内重复包含”。
- 单一定义规则关注的是“程序中实体定义是否重复”。
- 两者有关联，但不是同一个问题。

> [!warning] 易错点
> `#ifndef/#define/#endif` 不能把头文件中的全局变量定义变成安全写法。普通全局变量定义仍然不要放进公共头文件。

## 正确的头文件与实现文件分工

更推荐的工程结构是：

```cpp
// counter.h
#ifndef COUNTER_H
#define COUNTER_H

extern int count;      // 声明
int next_count();      // 声明

#endif
```

```cpp
// counter.cpp
#include "counter.h"

int count = 0;         // 定义

int next_count() {     // 定义
    return ++count;
}
```

其他 `.cpp` 使用时只包含头文件：

```cpp
// main.cpp
#include "counter.h"

int main() {
    return next_count();
}
```

这样每个 `.cpp` 都能通过头文件看到声明，而真正的定义只在 `counter.cpp` 中出现一次。

## 变量定义与初始化

变量定义的一般结构可以理解为：

```cpp
存储说明 类型 变量名 初始化部分;
```

常见写法：

```cpp
int age;        // 定义，未显式初始化
int value = 10; // 定义并用等号初始化
int score{90};  // 定义并用花括号初始化
```

课程强调：不要把多个变量定义挤在一行里，尤其是类型变复杂后容易误读。

不推荐：

```cpp
int i, j, k;
```

更清晰：

```cpp
int i = 0;
int j = 0;
int k = 0;
```

## 初始化顺序

在同一个作用域中，变量按代码出现顺序依次定义和初始化。

```cpp
int weight = 10;
int value = weight;
```

`value` 初始化时，前面的 `weight` 已经存在，因此 `value` 可以得到 `10`。

反过来则不行：

```cpp
int value = weight; // 错误：这里 weight 还没有声明
int weight = 10;
```

这和上一节的[[先声明后使用]]直接相关。

## 花括号初始化

C++11 以后可以使用[[花括号初始化]]：

```cpp
int value{10};
```

对于简单整数，`int value = 10;` 和 `int value{10};` 看起来差别不大；但对于复杂对象、数组或聚合类型，花括号初始化会更统一。

```cpp
int values[3]{1, 2, 3};
```

> [!tip] 初学者理解
> 先把花括号初始化理解为“另一种初始化语法”。后面学习类对象和构造函数时，它的价值会更明显。

## static 的两个常见语境

本节还提到 `static` 变量定义的两种常见含义。

### 静态局部变量

写在函数内部：

```cpp
void f() {
    static int count = 0;
    ++count;
}
```

这里的 `count` 是[[静态局部变量]]。它的作用域仍在**函数内部**，但生命周期不是每次函数调用开始创建、结束销毁，而是持续到程序结束。

### 文件作用域的 static 全局变量

写在所有函数外部：

```cpp
static int count = 0;
```

这里的 `count` 是文件作用域变量，并具有[[内部连接]]，只在当前编译单元内部可见。

这意味着其他 `.cpp` 即使也写了同名的 `static int count = 0;`，它们也是各自文件内部的不同变量。

## auto 的含义变化

早期 C/C++ 中，`auto` 表示[[自动变量]]，也就是普通局部变量默认具有的自动存储期。由于局部变量默认就是自动变量，实际代码中很少显式写：

```cpp
auto int a; // 旧式含义，现代 C++ 不再这样使用
```

C++11 以后，[[auto]] 的主要含义变成[[类型推导]]：

```cpp
auto value = 10;      // value 推导为 int
auto price = 12.5;    // price 推导为 double
```

编译器根据右侧初始化表达式的类型，在编译期确定变量类型。

> [!important] 现代用法
> 学现代 C++ 时，看到 `auto` 应优先理解为“让编译器自动推导类型”，而不是早期的“自动变量”存储说明。

## 本节速记

> [!summary] 速记
> 头文件主要放声明，实现文件放定义。声明可以重复，定义不能乱重复。包含警戒防的是同一编译单元内重复展开，不防多个 `.cpp` 各自生成一份定义。`extern` 是共享全局变量的声明入口，`static` 在文件作用域下会把名字限制在当前编译单元内部，现代 `auto` 主要用于类型推导。
