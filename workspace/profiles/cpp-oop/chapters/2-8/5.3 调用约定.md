# 5.3 调用约定

## 本节核心

[[调用约定]]规定函数底层如何被调用，影响参数传递、清栈责任以及函数符号名。

- 常见调用约定包括[[标准C调用]]、[[标准调用]]、[[快速调用]]、[[thiscall]] 等。
- 自己写的小程序通常使用编译器默认调用约定即可。
- 调用约定在跨语言、跨模块、动态链接库接口中很重要。
- 不同调用约定可能导致内部函数名表示不同。
- 成员函数通常隐含一个 `this` 指针，相关调用约定会围绕它处理。

> [!important] 核心认识
> 调用约定不是给人看的语法装饰，而是不同程序、不同语言、不同编译器之间正确互相调用的底层协议。

## 常见调用约定

| 调用约定 | 课程中的理解重点 |
|---|---|
| 标准 C 调用 | C/C++ 早期常见默认方式，支持可变参数 |
| 标准调用 | Windows API、动态链接库接口中常见 |
| Pascal 调用 | 历史调用方式，了解即可 |
| 快速调用 | 尽量把前几个参数放寄存器，提高调用效率 |
| thiscall | 成员函数常用，隐含 `this` 指针 |

普通练习程序中，通常不需要手动指定；但开发库、调用系统 API、跨语言调用时需要关注。

## 为什么调用约定重要

如果两个程序互相调用函数，调用方必须知道被调用函数的底层接口规则：

- 函数名在目标文件里叫什么。
- 参数按什么顺序传递。
- 参数放栈上还是寄存器里。
- 谁负责清栈。

如果这些规则不一致，调用方可能找不到函数，或者调用后栈被破坏。

## 函数名的内部表示

程序员写的函数名不一定就是目标文件里的符号名。

例如在某些调用约定和编译器下，函数名可能被加前缀、变大写或加入特殊符号。不同编译器也可能采用不同规则。

这就是为什么跨模块调用时不能只看源代码中的函数名，还要考虑调用约定和符号导出规则。

## 标准调用的应用场景

课程强调，Windows API 和动态链接库接口中常见[[标准调用]]。

当我们用 C++ 开发动态链接库并希望其他程序调用时，接口函数通常需要明确调用约定，保证外部程序知道怎么传参、怎么找函数、怎么清栈。

## thiscall 与成员函数

成员函数和普通函数不同，它要知道自己作用在哪个对象上。

成员函数通常隐含一个 `this` 指针：

```cpp
object.method();
```

底层可以理解为：调用 `method` 时还传入了 `object` 的地址，也就是 `this`。

[[thiscall]] 就是围绕成员函数和 `this` 指针形成的调用约定。

## 本节速记

> [!summary] 速记
> 调用约定是函数调用的底层协议。普通练习先用默认即可；写动态库、调系统 API、跨语言调用时必须关心。它会影响参数怎么传、谁清栈、函数符号怎么命名。成员函数通常隐含 `this` 指针。
