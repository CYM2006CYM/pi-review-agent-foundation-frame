# 6.1 类型的抽象

## 本节核心

进入类和对象之前，必须先理解：类不是从代码开始的，而是从 [[抽象]] 开始的。

面向对象程序设计中，一个类型可以看成一个 [[抽象数据类型]]：

> 数学模型 + 可施加的操作集合

换成更容易理解的话：

- 它是什么。
- 它有哪些数据。
- 数据之间满足什么关系。
- 它能做什么。

这些内容先形成抽象，再映射到 C++ 的 [[类]]。

## 抽象数据类型的基本组成

一个 [[抽象数据类型]] 通常包含三部分：

1. 名字
2. 数据及数据关系
3. 操作或行为

例如可以抽象一个“书”：

- 名字：书
- 数据：书名、页数、当前页、批注、书签
- 行为：翻到第一页、翻到指定页、读若干页、添加书签、添加批注

也可以抽象一个“水杯”：

- 名字：水杯
- 数据：容量、当前水量、温度、材质
- 行为：倒入水、喝掉水、倒掉水

抽象不是把现实事物完整复制到程序里，而是根据问题选择需要保留的部分。

## 抽象与具体表示无关

[[抽象数据类型]] 与具体表示方式无关。

你可以用文字描述，可以用表格描述，也可以用图描述。它不要求一定是某种代码形式。

例如“书”这个抽象，不一定一开始就写成：

```cpp
class Book {
    // ...
};
```

在写代码前，你完全可以先用自然语言描述：

- 一本书可以返回书名。
- 一本书可以翻页。
- 一本书可以添加书签。
- 一本书记录当前页。

这些描述已经是在做类型抽象。

> [!important] 关键观点
> 抽象阶段关注“这个事物在问题中有什么意义”，不是先关注“代码怎么写”。

## 抽象与现实存在无关

抽象出来的类型不一定必须对应现实中真实存在的东西。

只要它对解决问题有用，就可以抽象。

例如在程序中可以有：

- 游戏中的“技能”
- 操作系统中的“进程”
- 编译器中的“语法树节点”
- 网络程序中的“请求”

它们未必都是现实中可以摸到的物体，但在问题域里很有用，就可以成为类型。

## 同一事物可以有不同抽象

同一个现实事物，在不同问题中可以抽象出完全不同的类型。

例如“人”：

如果关注生命周期，可以抽象为：

- 数据：年龄
- 行为：出生、成长、死亡

如果关注学习系统，可以抽象为：

- 数据：知识水平、疲劳程度
- 行为：学习、睡觉、玩

同一个“地球”也一样。

如果研究国家经济，可以关注：

- 人口
- GDP
- 财富
- 统计总人口
- 计算总 GDP

如果研究天体运行，可以关注：

- 轨道位置
- 与太阳距离
- 线速度
- 与其他行星的位置关系

因此，抽象结果取决于 [[问题域]] 和关注点。

## 行为比数据更能体现类型

日常生活中，我们区分事物，往往不是只看数据，而是看行为。

一本很薄的书仍然是书，一份很厚的报纸仍然是报纸。真正帮助我们区分它们的，不只是页数，而是它们在使用场景中的行为。

在面向对象设计中也一样：一个类型的核心不是“里面有多少数据成员”，而是“它对外表现出什么行为”。

例如书的行为可以包括：

- 返回书名
- 翻到第一页
- 翻到指定页
- 阅读若干页
- 添加书签
- 添加批注
- 合上书

但在某个具体程序里，不一定全部需要。

## 抽象不要求面面俱到

抽象不是完整复刻现实世界。

正确的抽象应满足：

- 与当前问题有关的行为要保留。
- 与当前问题无关的行为可以忽略。
- 与当前问题有关的数据要保留。
- 与当前问题无关的数据不必加入。

例如做阅读器程序时，书的“卖给某个人”可能不需要；做图书销售系统时，这个行为就可能很重要。

> [!tip] 初学者判断方法
> 先问：“这个程序要解决什么问题？”再问：“为解决这个问题，这个对象必须知道什么、必须能做什么？”

## 从抽象走向类

本节先讨论抽象，下一节开始把抽象映射到 C++：

- 抽象数据类型的名字 -> [[类名]]
- 行为 -> [[成员函数]]
- 数据及关系 -> [[数据成员]]

这个过程就是从问题域模型走向程序语言表示。

## 本节考点整理

- [[抽象数据类型]] = 数学模型 + 操作集合。
- 抽象数据类型与具体表示无关，可以先用自然语言描述。
- 抽象出来的类型不一定必须现实存在，只要对问题有用即可。
- 同一事物在不同 [[问题域]] 中可以有不同抽象。
- 抽象时不要求面面俱到，只抽取解决问题需要的行为和数据。
- 面向对象设计中应优先关注对象的 [[行为]]，再考虑数据表示。

## 本节速记

> 类型不是代码里凭空写出来的，  
> 而是从问题中抽象出来的。  
> 关注点不同，抽象就不同。
