# 10.6 运算符重载的方针

## 本节核心

运算符重载不只是语法问题，更是类设计问题。

核心方针：

- 保持内置类型的使用习惯。
- 不能破坏运算符原本的直觉语义。
- 根据左操作数类型选择成员或非成员形式。
- 不要重载容易造成误解的运算符。

## 成员还是非成员

课堂给出的经验表可以整理为：

| 运算符类型 | 建议形式 |
|---|---|
| 所有一元运算符 | 成员函数 |
| `=`、`[]`、`()`、`->` | 必须是成员函数 |
| 复合赋值如 `+=`、`-=`、`*=` | 成员函数 |
| 其他普通二元运算符 | 非成员函数更常见 |

## 为什么一元运算符建议成员函数

一元运算符只有一个操作数。

如果重载发生在自定义类型上，这个操作数通常就是当前对象。

例如：

```cpp
++a;
```

写成成员函数很自然：

```cpp
a.operator++();
```

## 哪些必须是成员函数

以下运算符重载必须写成成员函数：

```cpp
operator=
operator[]
operator()
operator->
```

也就是：

- 赋值运算符。
- 下标运算符。
- 函数调用运算符。
- 箭头运算符。

> [!important] 高频考点
> `=`、`[]`、`()`、`->` 的重载必须是成员函数。这是选择题、填空题常见考点。

## 复合赋值建议成员函数

复合赋值类运算符左侧对象一定会被修改：

```cpp
a += b;
a -= b;
a *= b;
```

其语义是“修改左侧对象并返回左侧对象”，所以适合写成成员函数：

```cpp
A& operator+=(const A& rhs);
```

这与 [[赋值函数]] 的设计思路一致。

## 普通二元运算符建议非成员函数

普通二元运算符通常不修改操作数：

```cpp
a + b
a == b
a < b
```

这些运算符可以写成非成员函数，使左右操作数更对称。

特别是当左操作数不一定是自定义类型时，非成员函数更灵活。

例如：

```cpp
Complex operator+(double lhs, const Complex& rhs);
Complex operator+(const Complex& lhs, double rhs);
```

如果只写成员函数，就很难支持左操作数为内置类型的形式。

## 不要重载 && 和 ||

逻辑与 `&&`、逻辑或 `||` 对内置类型有短路求值特性：

```cpp
if (p != nullptr && p->valid()) {
}
```

如果重载 `&&` 或 `||`，它们会像普通函数调用一样先计算参数，无法保持内置逻辑运算符的短路语义。

因此通常不建议重载：

```cpp
operator&&
operator||
```

## 不要重载逗号运算符

逗号运算符 `,` 对内置类型也有固定求值习惯。

重载后容易让代码难读、语义不清晰，因此也不建议重载。

## 保持内置类型的使用习惯

运算符重载最重要的设计原则是：让自定义类型像内置类型一样自然，而不是制造惊讶。

例如：

- `+` 应该表示不改变原对象并产生新结果。
- `+=` 应该表示修改左侧对象。
- `==` 应该表示相等比较。
- `[]` 应该表示元素访问。
- `<<` 用于输出时，应返回输出流引用。

如果一个重载让读者无法凭符号猜出含义，就不应该使用运算符重载。

## 本节考点整理

- 运算符重载应保持内置类型的语义习惯。
- 一元运算符通常建议写成成员函数。
- `=`、`[]`、`()`、`->` 必须写成成员函数。
- 复合赋值运算符通常建议写成成员函数。
- 普通二元运算符通常建议写成非成员函数。
- `&&`、`||`、`,` 通常不建议重载。
- 不要通过运算符重载制造反直觉语义。
