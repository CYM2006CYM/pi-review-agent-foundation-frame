# 11.2 静态存储及其不足

## 本节核心

[[静态内存管理]] 的好处是自动、确定、容易理解；不足是空间大小和生命周期不够灵活。

本节通过全局对象、静态变量、局部变量和递归函数说明不同对象在内存中的位置。

## 数据区中的对象

以下对象通常存放在数据区：

```cpp
int total;
A obj1;
const A obj2;

class A {
public:
    static int base;
};
```

对应关系：

- `total` 是全局变量，位于全局数据区。
- `obj1` 是全局对象，位于全局数据区。
- `obj2` 是全局常对象，常放在常量数据区或只读相关区域。
- `A::base` 是静态数据成员，也在数据区。

即使 `base` 属于类，也不代表它存放在某个对象内部。静态数据成员属于类整体，有独立存储空间。

## 局部 static 变量

局部静态变量写在函数内部，但它不在栈区：

```cpp
void f() {
    static int count = 0;
    ++count;
}
```

`count` 的作用域在函数内部，但生命周期贯穿程序运行期间，所以通常存放在数据区。

> [!warning] 易错点
> “写在函数里面”不一定在栈区。带 `static` 的局部变量通常在数据区。

## 栈区中的对象

普通局部变量和普通局部对象通常位于栈区：

```cpp
int main() {
    int count = 0;
    A myObj;
}
```

函数调用时创建它们；函数返回时自动销毁它们。

递归函数会反复创建新的调用帧：

```cpp
int A::Func(int n) {
    if (n == 0) {
        return 0;
    }

    int local = n;
    return local + Func(n - 1);
}
```

每次递归调用都有自己的 `local`，因为每次调用都有独立栈帧。

## 静态存储的优点

静态/自动管理方式的优点：

- 不需要程序员手动释放普通局部对象。
- 离开作用域自动析构，管理简单。
- 全局对象、静态对象生命周期明确。
- 对小规模、固定大小数据非常方便。

例如：

```cpp
void f() {
    A obj;
} // obj 自动析构
```

## 静态存储的不足

静态存储的不足主要体现在灵活性上。

第一，对象数量有时无法提前确定。

```cpp
MyClass objs[50];
```

如果实际只需要 10 个，浪费空间；如果需要 100 个，又不够用。

第二，对象大小或数组规模可能要到运行时才知道。

编译阶段固定数组大小，会限制程序处理可变规模数据的能力。

第三，生命周期不够灵活。

栈区对象跟作用域绑定。离开函数就销毁，不能让它自然延续到函数外继续存在。

第四，栈空间有限。

大数组、大对象或深递归可能导致栈空间不足。

## 为什么引入动态内存管理

动态内存管理可以弥补上述不足。

通过 [[new运算符]]，程序可以在运行时按需要创建对象：

```cpp
A* p = new A;
```

通过 [[delete运算符]]，程序可以在合适时机释放对象：

```cpp
delete p;
```

这让对象数量、对象生命周期、资源释放时机都可以由程序逻辑控制。

## 本节考点整理

- 全局变量、全局对象、静态数据成员通常在数据区。
- 局部 `static` 变量虽然写在函数内，也通常在数据区。
- 普通局部变量和普通局部对象通常在栈区。
- 函数递归会产生多个栈帧。
- 静态/自动存储管理简单，但对象数量、大小、生命周期不够灵活。
- 动态内存管理用于在运行期间按需要创建和释放对象。
