# 12.2 名字空间的引入

这一节说明为什么需要[[名字空间]]。核心问题是：多人、多文件、多模块开发时，函数名、类名、变量名很容易重复，单靠人工改名并不可靠。

## 命名冲突从哪里来

假设一个项目由不同开发者共同完成。张三写了 `zhang.h` 和 `zhang.cpp`，李四写了 `li.h` 和 `li.cpp`。两个人可能都写出下面这样的名字：

```cpp
void f();

class T {
public:
    void ZhangFunc();
};
```

另一个文件里也可能有：

```cpp
void f();

class T {
public:
    void LiFunc();
};
```

如果这些名字都放在同一个全局范围中，链接或编译时就可能出现[[命名冲突]]：两个 `f`、两个 `T` 到底指谁？

## 为什么不能只靠改名

表面上看，冲突了就改名即可。但在真实项目中，改名会带来几个问题：

- 已经写好的代码可能很多，改名会牵动大量调用处。
- 多人协作时，沟通成本高，不能保证所有人同步修改。
- 代码可能来自第三方库或历史模块，无法随意要求对方改名。
- 改名本身可能引入新的错误。

因此需要一种机制，让不同模块可以保留自己的自然命名，同时又不互相冲突。

## 名字空间的作用

[[名字空间]]就是为名字增加一层“归属范围”。

```cpp
namespace Zhang {
    void f();

    class T {
    public:
        void ZhangFunc();
    };
}

namespace Li {
    void f();

    class T {
    public:
        void LiFunc();
    };
}
```

这样 `Zhang::f` 和 `Li::f` 是两个不同的名字，`Zhang::T` 和 `Li::T` 也是两个不同的类型。它们在各自名字空间中可以同名，但在完整限定名上并不相同。

> [!important] 高频考点
> 名字空间解决的是“名字归属”和“命名冲突”问题，不改变函数、类、变量本身的语义。

## 和包含警戒的区别

[[包含警戒]]解决的是同一个头文件被重复包含的问题；[[名字空间]]解决的是不同代码单元中名字重复的问题。

| 机制 | 解决的问题 |
|---|---|
| 包含警戒 | 避免同一头文件内容被重复展开 |
| 名字空间 | 避免不同模块中的同名实体冲突 |

二者经常同时出现在头文件中，但用途不同。

## 本节小结

- 大型项目中，同名函数、类、变量很常见。
- 直接改名成本高，也容易引入新错误。
- [[名字空间]]用“名字归属范围”减少冲突。
- `Zhang::T` 和 `Li::T` 可以同时存在，因为完整名字不同。
