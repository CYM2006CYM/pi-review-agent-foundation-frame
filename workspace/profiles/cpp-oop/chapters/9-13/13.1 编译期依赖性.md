# 13.1 编译期依赖性

前面章节主要讨论单个类的定义、实现、构造、析构、拷贝、赋值和运算符重载。真实程序通常不是一个类单独工作，而是多个类实例化为多个对象，通过交互完成整体功能。

类之间的联系可以先分成两层：

- [[物理关系]]：源文件、头文件之间的包含和编译依赖。
- [[逻辑关系]]：类在问题域和设计模型中的语义关系。

本节重点讨论物理层面的[[编译期依赖]]。

## 单向联系与双向联系

类之间可以是[[单向联系]]，也可以是[[双向联系]]。

```cpp
class A {
};

class B {
public:
    void f(A& a);

private:
    A* pA;
};
```

这里 `B` 使用了 `A`，可以说 `B` 指向或依赖 `A`。如果 `A` 并不使用 `B`，这种关系就是单向的。

双向联系则是双方都使用对方：

```cpp
class A {
    B* pB;
};

class B {
    A* pA;
};
```

双向联系通常比单向联系复杂，维护成本也更高。设计时应优先把关系设计成单向。

## 物理关系：头文件与实现文件

一个类通常拆成两个文件：

- `.h`：放类声明、接口、必要的类型信息。
- `.cpp`：放成员函数实现。

当一个头文件必须 `#include` 另一个头文件时，两个文件之间的编译期依赖就变强。依赖越强，一个文件改动后引起的重新编译范围越大，也越不利于扩展和维护。

## 对象成员会增强依赖

如果 `B` 中直接包含 `A` 类型的对象成员：

```cpp
// B.h
#include "A.h"

class B {
private:
    A mA;
};
```

编译器在确定 `B` 的对象大小时，必须知道 `A` 的完整定义。因此这里只写[[前置声明]]不够：

```cpp
class A;

class B {
private:
    A mA; // 错误：不知道 A 的大小
};
```

所以对象成员会迫使 `B.h` 包含 `A.h`，从而增强文件间依赖。

> [!warning] 易错点
> 前置声明只告诉编译器“有这样一个类”，不能告诉编译器这个类的大小、成员函数和布局。

## 内联实现也会增强依赖

如果成员函数在头文件中内联实现，并且函数体调用了 `A` 的成员函数：

```cpp
// B.h
#include "A.h"

class B {
public:
    int f(A* pA) {
        return pA->g();
    }
};
```

这里编译器不仅要知道 `A` 是一个类，还要知道 `A` 是否有 `g()`。因此必须包含 `A.h`。

如果把函数体放到 `.cpp` 中：

```cpp
// B.h
class A;

class B {
public:
    int f(A* pA);
};
```

```cpp
// B.cpp
#include "B.h"
#include "A.h"

int B::f(A* pA)
{
    return pA->g();
}
```

这样 `B.h` 只需要知道 `A` 是个类，完整定义放到 `B.cpp` 中再包含。[[外联实现]]可以降低头文件之间的依赖。

## 关联强弱层次

从物理依赖强弱看，可以粗略理解为：

1. [[继承]]：最强，后续章节会讲。
2. [[硬关联]]：双向联系且存在强关联，最不利于维护。
3. [[强关联]]：头文件必须包含另一个头文件。
4. [[弱关联]]：实现文件包含其他头文件，头文件中尽量前置声明。
5. [[软关联]]：尽量只通过指针或引用发生联系。

设计时应有意识地降低编译期依赖，尽可能选择更弱的关系。

## 为什么常用指针和引用

如果成员中只保存指针或引用：

```cpp
class A;

class B {
private:
    A* pA;
};
```

编译器只需要知道 `A` 是一个类，因为指针大小固定，不需要知道 `A` 的完整对象大小。这就是[[软关联]]常见的形式。

函数参数也类似：

```cpp
void f(const A& a);
void g(A* p);
```

相比按值传递：

```cpp
void h(A a);
```

引用和指针通常更能降低依赖，也避免不必要的[[拷贝构造函数]]调用。

指针和引用选择的一般原则：

- 参数必须存在，不允许为空：优先用引用。
- 参数可有可无，允许为空：优先用指针。

## 本节小结

- 多类程序既有物理关系，也有逻辑关系。
- 物理关系主要体现为头文件和实现文件之间的包含关系。
- 对象成员需要完整类型定义，会增强编译期依赖。
- 内联实现如果调用其他类成员，也会增强头文件依赖。
- [[前置声明]]加[[外联实现]]可以降低依赖。
- 设计时应尽量降低从硬关联到软关联的物理耦合。
