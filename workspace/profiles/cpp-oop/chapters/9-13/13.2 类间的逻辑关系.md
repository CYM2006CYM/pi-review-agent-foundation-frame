# 13.2 类间的逻辑关系

上一节讨论类之间的物理关系，也就是文件和编译依赖。这一节讨论[[逻辑关系]]：从设计语义上看，两个类之间到底是什么关系。

## 垂直方向与水平方向

类间逻辑关系可以先按方向分成两类：

| 方向 | 典型关系 | 本阶段是否展开 |
|---|---|---|
| 垂直方向 | 泛化、实现、继承 | 后续继承章节展开 |
| 水平方向 | [[关联]]、[[依赖]] | 本章重点 |

垂直方向通常对应“父类和子类”“抽象与具体”等继承相关关系。当前阶段先讨论没有继承的平等类之间如何发生联系，也就是水平方向关系。

## 水平方向的代码形式

两个类在水平方向发生联系，常见代码形式有四种。

### 1. 数据成员形式

```cpp
class B {
private:
    A* pA;
};
```

`B` 中有一个 `A` 类型或指向 `A` 的数据成员。这表示 `B` 对象长期持有或记录与 `A` 对象的关系。

这种形式称为[[关联]]。

### 2. 函数参数形式

```cpp
class B {
public:
    void func(A* pA);
};
```

`A` 只作为某个函数调用时传入的参数出现。

这种关系通常称为[[依赖]]。

### 3. 函数返回值形式

```cpp
class B {
public:
    A* func01();
};
```

`B` 的某个函数返回 `A` 类型或 `A` 指针/引用。它也说明函数层面用到了 `A`。

这种关系也属于[[依赖]]。

### 4. 函数实现形式

```cpp
int B::func02(int n)
{
    A* pA = new A;
    int val = pA->g() + n;
    delete pA;
    return val;
}
```

函数声明看不出和 `A` 有关，但函数实现内部创建或使用了 `A`。这也是一种依赖。

## 关联和依赖的区别

核心区别在于关系强弱。

| 关系 | 代码表现 | 强弱 |
|---|---|---|
| [[关联]] | `A` 是 `B` 的数据成员 | 较强 |
| [[依赖]] | `A` 出现在函数参数、返回值或实现中 | 较弱 |

如果 `B` 中有 `A` 的数据成员，那么创建 `B` 对象时，`B` 对 `A` 的关系长期存在。即使是指针成员，也表示 `B` 结构中保留了与 `A` 的联系。

如果 `A` 只出现在函数参数、返回值或函数体中，`B` 对 `A` 的使用可能只发生在某次调用中。创建 `B` 对象时，不一定立即需要 `A` 对象。

> [!important] 高频考点
> 数据成员形式是关联；函数参数、函数返回值、函数实现内部使用，通常归为依赖。

## 和 UML 表示的对应

在 UML 中，类之间联系可以用线条和箭头表达：

- 单向联系：带方向箭头。
- 双向联系：无方向或双方可导航。
- 关联：通常用实线表示。
- 依赖：通常用虚线箭头表示。

考试更常要求能从代码判断关系，而不是记图形符号本身。

## 本节小结

- 类间逻辑关系分垂直方向和水平方向。
- 垂直方向主要是继承相关关系，后续再讲。
- 水平方向本章重点是[[关联]]和[[依赖]]。
- 数据成员形式表示关联。
- 函数参数、返回值、函数实现内部使用表示依赖。
- 关联通常比依赖关系更强。
