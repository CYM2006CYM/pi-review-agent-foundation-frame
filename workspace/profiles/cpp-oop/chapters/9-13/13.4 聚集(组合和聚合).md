# 13.4 聚集(组合和聚合)

[[聚集关联]] 是一种特殊的[[关联]]。它不仅表示一个对象知道另一个对象，还强调二者是整体和部分的关系。

聚集关联分成两类：

| 类型 | 英文 | 核心含义 |
|---|---|---|
| [[聚合]] | Aggregation | 整体拥有或容纳部分，但不控制部分生命周期 |
| [[组合]] | Composition | 整体拥有部分，并控制部分生命周期 |

## 聚合

[[聚合]] 表示整体和部分之间存在关系，但整体不负责部分的创建与销毁。

例如计算机和网卡：

- 计算机可以没有网卡。
- 网卡也可以脱离某台计算机而存在。
- 计算机不一定负责网卡对象的创建与销毁。

代码上常见形式是：整体保存外部传入的部分对象指针，但析构时不删除它。

```cpp
class A;

class B {
public:
    B(A* pa) : mpA(pa) {}
    ~B() = default;

private:
    A* mpA;
};
```

这里 `B` 知道 `A`，但 `A` 的生存不由 `B` 控制。`B` 对象消失时，`A` 对象不一定消失。

## 组合

[[组合]] 表示更强的整体-部分关系。整体负责部分对象的创建与销毁，部分的生命周期受整体控制。

```cpp
class A;

class B {
public:
    B(int n) : mpA(new A(n)) {}

    ~B()
    {
        delete mpA;
    }

private:
    A* mpA;
};
```

这里 `B` 在构造时创建 `A`，在析构时释放 `A`，因此 `A` 的生命周期由 `B` 管理。这就是组合。

> [!warning] 易错点
> 组合不一定要求整体和部分在同一行代码中“同时出生、同时死亡”。关键不是时间完全同步，而是整体是否拥有并控制部分的创建、释放和存在资格。

## UML 表示

| 关系 | UML 常见符号 | 菱形所在端 |
|---|---|---|
| 聚合 | 空心菱形 | 整体一端 |
| 组合 | 实心菱形 | 整体一端 |

考试中更常考语义判断和代码判断：如果只保存外部对象而不负责释放，偏向聚合；如果内部创建并负责释放，偏向组合。

## 不要机械照搬现实世界

类间关系不是把现实关系原样搬进程序。现实中汽车和轮子看起来像组合，但在程序中未必总是组合。

如果系统是汽车制造模拟，车对象可能创建并销毁轮子对象，适合组合。  
如果系统是维修库存管理，轮子可能独立入库、调拨、装配，适合聚合或一般关联。

> [!important] 高频考点
> 聚合和组合的分界不在“现实中谁属于谁”，而在“程序模型中谁负责谁的生命周期”。

## 本节小结

- 聚集关联强调整体-部分关系。
- 聚合：整体包含部分，但不控制部分生命周期。
- 组合：整体包含部分，并控制部分生命周期。
- 同一现实关系在不同问题域中可能建模成不同逻辑关系。
- 代码上是否 `new`、是否 `delete`、是否拥有对象，是判断生命周期责任的重要线索。
