# 13.5 依赖

[[依赖]] 是比[[关联]]更弱的类间逻辑关系。它表示一个类在某个局部行为中使用另一个类，但不把对方长期保存为自己的成员。

常见代码形式有三种：

| 形式 | 示例 | 含义 |
|---|---|---|
| 函数参数 | `void f(A* pa);` | 调用某个行为时需要 `A` |
| 函数返回值 | `A* g();` | 行为结果涉及 `A` |
| 函数实现内部使用 | `A* p = new A;` | 只在函数体内部临时使用 `A` |

UML 中，依赖通常用虚线箭头表示，箭头从使用者指向被使用者。

## 依赖为什么更弱

如果 `B` 把 `A` 作为数据成员保存，`B` 对象的结构中长期包含 `A` 的位置，这是关联。

如果 `B` 只在某个函数里接收、返回或临时创建 `A`，`B` 对象本身不需要长期知道 `A`。这种联系只在行为发生时出现，因此更弱。

> [!important] 高频考点
> 如果一种关系可以设计成依赖，就不要轻易设计成关联。依赖更弱，类之间耦合更低，维护成本通常更小。

## 老鼠吃苹果的例子

考虑“老鼠吃苹果”。

- 老鼠不需要一直知道所有苹果。
- 苹果也不需要一直知道所有老鼠。
- “吃”是一个行为，行为发生时才需要把老鼠和苹果联系起来。

因此，这个模型更适合设计成依赖，而不是关联。

```cpp
class Apple {
public:
    explicit Apple(int power) : power(power) {}

    int getEnergy() const
    {
        return power;
    }

private:
    int power;
};

class Mouse {
public:
    explicit Mouse(int weight) : weight(weight) {}

    int getWeight() const
    {
        return weight;
    }

    void eat(Apple* one)
    {
        if (one != nullptr) {
            weight += static_cast<int>(one->getEnergy() * 0.5);
        }
    }

private:
    int weight;
};
```

这里 `Mouse::eat` 的参数是 `Apple*`，说明 `Mouse` 在吃这个行为中依赖 `Apple`。但 `Mouse` 没有保存 `Apple*` 成员，所以不是关联。

## 行为应该放在哪个类里

一个行为常常涉及多个对象。设计时要判断谁是主动方，谁是被动方。

在“老鼠吃苹果”中，老鼠是主动方，苹果是被动方。因此常见设计是把 `eat` 放在 `Mouse` 中，把 `Apple*` 作为参数传入。

这体现了[[行为归属]]原则：

- 行为尽量放在主动、复杂、变化较多的一方。
- 被使用对象作为参数传入即可。
- 如果两边都需要提供接口，也应避免把核心逻辑重复写两遍。

例如也可以让苹果提供被吃的接口：

```cpp
class Mouse;

class Apple {
public:
    int eatenBy(Mouse* m);
};

class Mouse {
public:
    void eat(Apple* one)
    {
        if (one != nullptr) {
            weight += one->eatenBy(this);
        }
    }

private:
    int weight;
};
```

这时 `Mouse::eat` 只是协调，真正能量计算可以集中在 `Apple::eatenBy`。关键是不要在两个类里各写一套互相不一致的核心规则。

## 本节小结

- 依赖是较弱的类间关系。
- 参数、返回值、函数体内部临时使用，通常是依赖。
- 依赖比关联耦合低，能用依赖就不要随意升级为关联。
- 行为涉及多个类时，要判断主动方和被动方。
- 重复实现同一行为规则容易造成维护错误，应通过委托集中核心逻辑。

2. 为什么“老鼠吃苹果”不适合建模成老鼠和苹果的长期关联？
3. 行为放在主动对象中有什么好处？
4. 如果两个类都暴露相关行为，为什么要避免重复实现核心逻辑？
