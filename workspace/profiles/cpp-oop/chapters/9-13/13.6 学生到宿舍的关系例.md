# 13.6 学生到宿舍的关系例

“学生和宿舍”是理解[[关联]]、[[聚合]]、[[组合]]的典型例子。现实中，学生和宿舍确实互相有关，但程序设计不能把现实关系原样搬进代码，必须看当前系统的关注点。

## 现实关系与程序关系

现实中：

- 学生通常知道自己住哪个宿舍。
- 宿舍也知道里面住了哪些学生。
- 学生毕业离开，宿舍不会随之消失。
- 宿舍存在多年，里面的学生会不断变化。

因此从现实直觉看，学生和宿舍通常是普通关联；如果强调整体-部分，也更接近[[聚合]]，不是[[组合]]。

但程序设计时还要看问题域：

| 程序类型 | 关注中心 | 可能的建模方向 |
|---|---|---|
| 通讯录 | 学生地址信息 | 学生知道宿舍 |
| 学生管理系统 | 学生档案完整性 | 学生到宿舍关系更重要 |
| 宿舍管理系统 | 床位和入住管理 | 宿舍到学生关系更重要 |

> [!important] 高频考点
> 类间关系不是现实关系的机械翻译，而是针对问题域建立的程序模型。

## 学生到宿舍：一般关联

如果只强调学生应该知道自己的地址信息，可以让 `Student` 保存 `Dorm*`。

```cpp
class Dorm;

class Student {
public:
    explicit Student(Dorm* aDorm) : mpDorm(aDorm) {}

    const char* address() const
    {
        if (mpDorm != nullptr) {
            return mpDorm->floor();
        }
        return "";
    }

private:
    Dorm* mpDorm;
};
```

这里学生对象知道宿舍对象，哪怕宿舍指针为空，也表示学生类中保留了宿舍信息的位置。这是从学生到宿舍的[[一般关联]]。

## 学生到宿舍：聚合

如果设计者强调“宿舍信息是学生信息的一部分”，但学生不负责宿舍对象的创建和销毁，就可以把它理解为[[聚合]]。

代码形式可能仍然和一般关联很像：

```cpp
class Student {
public:
    explicit Student(Dorm* aDorm) : mpDorm(aDorm) {}

private:
    Dorm* mpDorm;
};
```

同样的代码形式可以表达不同逻辑关系，因为逻辑关系存在于设计者的建模意图中。代码只提供线索，不能完全替代设计语义。

> [!warning] 易错点
> 看到 `Dorm* mpDorm` 只能判断“有较强联系”，不能仅凭这一行断定一定是一般关联、聚合或组合。还要看设计语义和生命周期责任。

## 学生到宿舍：组合

如果模型规定学生对象创建时必须创建自己的宿舍信息，并且学生对象销毁时也负责释放这份宿舍信息，那么可以设计成[[组合]]。

```cpp
class Student {
public:
    Student() : mpDorm(new Dorm) {}

    ~Student()
    {
        delete mpDorm;
    }

private:
    Dorm* mpDorm;
};
```

这时宿舍信息被视为学生对象的必要组成部分，且生命周期由学生控制，所以是组合。

## 本节小结

- 学生到宿舍的方向可以建模成一般关联、聚合或组合。
- 决定关系类型的不是现实关系本身，而是程序关注点。
- 一般关联强调学生“知道”宿舍。
- 聚合强调宿舍信息是学生信息的一部分，但学生不控制宿舍生命周期。
- 组合强调学生拥有并控制宿舍信息的生命周期。

## 复习检查

1. 为什么学生和宿舍的现实关系不能直接决定程序关系？
2. 学生保存 `Dorm*` 时，为什么可能是一般关联也可能是聚合？
3. 什么时候学生到宿舍可以建模成组合？
4. 代码形式和设计语义发生冲突时，判断关系应优先看什么？
