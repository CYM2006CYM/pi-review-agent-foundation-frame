# 9.4 使用自定义拷贝构造函数

## 本节核心

当 [[缺省拷贝构造函数]] 的 [[浅拷贝]] 不符合类的语义时，就需要自定义 [[拷贝构造函数]]。

最常见的原因是：类中含有指针成员，并且对象负责释放指针指向的资源。

此时缺省拷贝只复制指针地址，容易导致两个对象共享同一块资源，析构时发生 [[重复释放]]。

> [!important] 高频考点
> 自定义拷贝构造函数的核心不是“机械复制所有成员”，而是按程序员定义的对象语义进行拷贝。需要独立资源副本时，应实现深拷贝；不允许复制的类，应明确禁止拷贝。

## 自定义拷贝构造用于深拷贝

上一节中的问题类大致如下：

```cpp
class B {
public:
    B() {
        pa_ = new A;
    }

    ~B() {
        delete pa_;
    }

private:
    A* pa_;
};
```

如果直接使用缺省拷贝构造函数：

```cpp
B b1;
B b2(b1);
```

`b1.pa_` 和 `b2.pa_` 会指向同一个堆区对象。

正确思路是：拷贝 `B` 时，为新对象重新创建一份独立的 `A` 对象。

```cpp
class B {
public:
    B(const B& other) {
        pa_ = new A(*other.pa_);
    }

private:
    A* pa_;
};
```

这样：

- `b1.pa_` 指向一个堆区 `A` 对象。
- `b2.pa_` 指向另一个堆区 `A` 对象。
- 两个 `A` 对象内容相同，但地址不同。
- 两个 `B` 对象析构时各自释放自己的资源。

这就是 [[深拷贝]]。

## 深拷贝不一定只有一种写法

拷贝构造函数可以在函数体中完成资源创建：

```cpp
B::B(const B& other) {
    pa_ = new A(*other.pa_);
}
```

也可以放在 [[构造函数初始化列表]] 中：

```cpp
B::B(const B& other)
    : pa_(new A(*other.pa_))
{
}
```

两种写法都能表达“为新对象分配独立资源”。

实际写法应优先考虑成员是否必须在初始化阶段完成构造。

## 拷贝构造函数仍然是构造函数

拷贝构造函数是构造函数的一种特殊情况，因此普通构造函数的规则仍然适用。

例如类中有引用成员：

```cpp
class B {
public:
    B(const B& other);

private:
    A& ra_;
};
```

引用成员必须在初始化列表中初始化，不能进入构造函数体后再赋值。

```cpp
B::B(const B& other)
    : ra_(other.ra_)
{
}
```

如果忘记在初始化列表中处理引用成员，编译器会报错。

类似地，`const` 成员、没有无参构造函数的对象成员，也常常必须在初始化列表中处理。

## 什么时候需要自定义拷贝构造函数

常见情况有三类：

- 需要 [[深拷贝]]。
- 需要按业务语义定制拷贝行为。
- 需要禁止对象被拷贝。

如果类只是普通值类型，成员也都能正确拷贝，缺省拷贝构造函数可能已经足够。

如果类负责管理资源，就应认真检查拷贝语义。

## 禁止拷贝：私有拷贝构造函数

早期 C++ 中常见做法是把拷贝构造函数声明为 `private`：

```cpp
class MyClass {
private:
    MyClass(const MyClass& other);
};
```

这样类外代码不能调用拷贝构造函数：

```cpp
MyClass a;
MyClass b(a); // 错误：拷贝构造函数是 private
```

同时，用户已经声明了拷贝构造函数，编译器也不会再提供缺省拷贝构造函数。

## 私有拷贝构造的限制

私有拷贝构造函数只能限制类外访问。

如果在本类成员函数内部调用：

```cpp
class MyClass {
public:
    void f() {
        MyClass a;
        MyClass b(a); // 本类内部可以访问 private
    }

private:
    MyClass(const MyClass& other);
};
```

只要这个私有函数有实现，这段代码在本类内部仍然可能成立。

因此，仅仅设为 `private` 不是绝对禁止拷贝。

## 禁止拷贝：只声明不实现

另一种旧式写法是只声明拷贝构造函数，但不提供实现：

```cpp
class MyClass {
public:
    MyClass(const MyClass& other);
};
```

这样编译器认为用户已经声明了拷贝构造函数，不再自动生成缺省版本。

如果代码实际调用它：

```cpp
MyClass a;
MyClass b(a);
```

编译阶段可能通过，但链接阶段找不到函数实现，会产生 [[链接错误]]。

这种方式能阻止实际生成可执行文件，但错误暴露得较晚。

## 编译错误优于链接错误

如果拷贝构造函数是私有的，类外调用时会产生 [[编译错误]]。

如果拷贝构造函数是公有声明但没有实现，调用时可能产生链接错误。

调试角度看：

- 编译错误更直接，定位更早。
- 链接错误更晚，通常比编译错误难排查。
- 运行时逻辑错误更晚，更难排查。

因此，能在编译期阻止的问题，尽量让它在编译期暴露。

## C++11 以后：使用 = delete

C++11 以后，可以用 [[delete函数]] 明确删除拷贝构造函数：

```cpp
class MyClass {
public:
    MyClass(const MyClass& other) = delete;
};
```

这表示：

> 这个函数被明确禁止使用。

任何试图调用拷贝构造函数的代码都会在编译期报错。

相比“私有且不实现”的旧式写法，`= delete` 更清晰、更直接。

## 禁止按值传参和按值返回

按值传参需要拷贝构造：

```cpp
void f(MyClass obj);
```

如果 `MyClass` 禁止拷贝，那么：

```cpp
MyClass a;
f(a); // 错误：需要拷贝构造
```

按值返回也可能需要拷贝构造：

```cpp
MyClass make();
```

因此，禁止拷贝不仅禁止 `MyClass b(a);`，也会限制按值传参和按值返回。

这类对象通常应使用引用、指针或其他资源管理方式传递。

## 本节考点整理

- 缺省拷贝构造函数可能产生 [[浅拷贝]] 问题。
- 有指针成员且析构时释放资源的类，通常需要自定义 [[拷贝构造函数]]。
- [[深拷贝]] 会为新对象创建独立资源副本。
- 拷贝构造函数仍是构造函数，必须遵守初始化规则。
- 引用成员、`const` 成员、对象成员可能必须在初始化列表中处理。
- 自定义拷贝构造可用于实现深拷贝。
- 自定义拷贝构造也可用于禁止拷贝。
- 私有拷贝构造函数可限制类外拷贝。
- 只声明不实现可能导致 [[链接错误]]。
- C++11 后推荐用 [[delete函数]] 禁止拷贝。
- 禁止拷贝会影响按值传参和按值返回。

## 本节速记

> 指针管资源，浅拷贝危险；  
> 要独立资源就深拷贝；  
> 不许拷贝就 `= delete`。
