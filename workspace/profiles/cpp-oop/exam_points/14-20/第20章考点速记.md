# 第20章考点速记

第20章重点不是语法，而是面向对象综合设计：如何识别变化，如何在[[依赖]]、[[关联]]、[[组合]]和[[继承]]之间选择关系。

## 高频判断

| 问题 | 优先判断 |
|---|---|
| A 临时使用 B 完成某个函数 | [[依赖]] |
| A 长期保存 B 的指针、引用或对象成员 | [[关联]] |
| A 是整体，B 是部分，但生命周期相对独立 | [[聚合]] |
| A 创建并销毁 B，B 是 A 的组成部分 | [[组合]] |
| B 是一种 A | [[公有继承]] |
| 只是为了复用实现 | 优先[[组合]]，谨慎继承 |

## 三个核心原则

1. [[封装变化]]：先找出可能变化的方向，再用合适结构隔离变化。
2. [[针对接口编程]]：依赖父类接口或抽象类，而不是依赖具体类。
3. [[优先组合]]：没有明确 `is-a` 关系时，不要轻易使用继承。

## 依赖和继承

典型形式：

```cpp
void A::f(Parent& p) {
    p.func();
}
```

`A` 通过参数依赖 `Parent`，实际对象可以是 `Parent` 的任何派生类。适合表达“一次调用中使用某对象”的关系。

常见例子：

- 老鼠吃水果；
- 学生做作业；
- 屏幕绘制区域；
- 怪物与怪物战斗；
- 鳄鱼攻击怪物。

## 关联和继承

典型形式：

```cpp
class A {
private:
    Parent* p;
};
```

`A` 长期关联 `Parent`，而 `Parent` 的实际对象可以由派生类扩展。适合表达“当前对象持有或长期使用另一个对象”的关系。

常见例子：

- 警察关联车辆；
- 科学家关联计算工具；
- 篮子组合水果；
- 学生关联同学；
- 公司组合部门。

## 考试易错点

- 不要把所有变化都塞进同一棵继承树，否则容易形成[[子类爆炸]]。
- 公有继承要求语义上成立 `is-a`，不是“想复用代码”就继承。
- `A` 中有 `B` 成员时，优先考虑 `has-a`、关联、组合，而不是继承。
- 关联父类接口时，新增派生类一般不应导致关联方修改。
- 公司-部门这类树形结构中，容器类也可以是父类的一种，从而形成[[递归组合结构]]。

## 一句话复习

面向对象设计的目标是让变化发生在局部：用[[继承]]承接同类对象的种类变化，用[[依赖]]和[[关联]]组织对象协作，用[[组合]]表达整体-部分结构。
