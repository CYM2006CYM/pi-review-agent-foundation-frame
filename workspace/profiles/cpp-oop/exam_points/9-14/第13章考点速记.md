# 第13章考点速记

## 一页总览

| 主题 | 必背结论 |
|---|---|
| [[编译期依赖]] | 头文件包含会形成物理依赖，尽量用前置声明降低依赖 |
| [[关联]] | 一个类把另一个类作为数据成员保存，关系较强 |
| [[依赖]] | 另一个类只出现在函数参数、返回值或函数体中，关系较弱 |
| [[一般关联]] | 强调长期、非偶然的“知道” |
| [[聚合]] | 整体包含部分，但不控制部分生命周期 |
| [[组合]] | 整体包含部分，并控制部分生命周期 |
| [[自关联]] | 一个类和自身发生关联，通常是同类不同对象之间的联系 |
| [[关联类]] | 当关系本身有属性时，把关系抽象成类 |

## 代码判断

```cpp
class B {
private:
    A* pA;
};
```

通常是关联。进一步是一般关联、聚合还是组合，要看语义和生命周期。

```cpp
class B {
public:
    void f(A* pA);
    A* g();
};
```

通常是依赖。

```cpp
B::B() : pA(new A) {}
B::~B() { delete pA; }
```

如果 `B` 被视为整体，`A` 被视为部分，并且 `B` 控制 `A` 生命周期，通常是组合。

```cpp
B::B(A* pa) : pA(pa) {}
B::~B() = default;
```

如果 `A` 外部创建，`B` 只保存它，通常是聚合或一般关联。

## 高频辨析

| 易混点 | 正确理解 |
|---|---|
| 关联 vs 依赖 | 数据成员是关联；参数、返回值、函数体内部临时使用是依赖 |
| 聚合 vs 组合 | 是否由整体负责部分对象的创建与销毁 |
| 代码形式 vs 逻辑语义 | 同一代码可能因设计意图不同表示不同逻辑关系 |
| 现实关系 vs 程序关系 | 程序关系由问题域和关注点决定，不机械照搬现实 |
| 解除联系 vs 删除对象 | 指针设为空只是解除联系，`delete` 才释放对象 |

## 学生与宿舍例题

从学生到宿舍：

- 学生只知道宿舍地址：一般关联。
- 宿舍信息是学生信息的一部分，但学生不创建宿舍：聚合。
- 学生创建并释放宿舍信息：组合。

从宿舍到学生：

- 宿舍只知道住了哪些学生：一般关联。
- 宿舍保存外部学生对象，不删除学生：聚合。
- 宿舍创建并删除学生对象：组合。
